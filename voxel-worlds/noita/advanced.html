💤 Sleeping Chunks: Skip the Settled World

Most of a voxel world is settled at any moment — sand pile already piled, walls inert, untouched air. Re-running the per-cell rule on those cells is wasted time. The fix: partition the grid into fixed chunks (we use 32×32), track an active counter per chunk, and step only chunks whose counter is non-zero.

The wake/sleep protocol

  1. Every cell movement wakes its chunk — counter = N (say 8 frames).
  2. Crucially, also wake the 8 neighbour chunks: motion crossing a chunk border must be visible to the chunk it enters.
  3. End of frame, decrement every chunk's counter. Zero → asleep next frame.
  4. Painting via mouse wakes the touched chunks too.

Result: a calm world drops to a handful of awake chunks; a busy one wakes only the busy region. Per-frame work scales with active area, not total area.

// World layout: grid + per-chunk activity counter.
function makeWorld(W, H) {
    const Wc = Math.ceil(W / CHUNK), Hc = Math.ceil(H / CHUNK);
    return { W, H, Wc, Hc,
             grid:        new Uint8Array(W * H),
             chunkActive: new Uint8Array(Wc * Hc).fill(WAKE_FRAMES) };
}

// Any cell change wakes a 3×3 chunk neighbourhood. Cheap, and catches motion
// crossing chunk borders.
function wakeAt(w, x, y) {
    const cx = Math.floor(x / CHUNK), cy = Math.floor(y / CHUNK);
    for (let ny = Math.max(0, cy - 1); ny <= Math.min(w.Hc - 1, cy + 1); ny++)
        for (let nx = Math.max(0, cx - 1); nx <= Math.min(w.Wc - 1, cx + 1); nx++)
            w.chunkActive[ny * w.Wc + nx] = WAKE_FRAMES;
}

function step(w, frame) {
    for (let cy = w.Hc - 1; cy >= 0; cy--) {
        for (let cx = 0; cx < w.Wc; cx++) {
            if (w.chunkActive[cy * w.Wc + cx] === 0) continue;   // skip sleeping
            // ... step every cell in [cx*CHUNK, cy*CHUNK) … same per-cell logic
            // as the Intermediate tier; every move calls wakeAt().
        }
    }
    // Tick down quiet chunks — eventually they sleep.
    for (let c = 0; c < w.chunkActive.length; c++)
        if (w.chunkActive[c] > 0) w.chunkActive[c]--;
}
type World = {
    W: number; H: number; Wc: number; Hc: number;
    grid: Uint8Array; chunkActive: Uint8Array;
};

function makeWorld(W: number, H: number): World {
    const Wc = Math.ceil(W / CHUNK), Hc = Math.ceil(H / CHUNK);
    return { W, H, Wc, Hc,
             grid:        new Uint8Array(W * H),
             chunkActive: new Uint8Array(Wc * Hc).fill(WAKE_FRAMES) };
}

function wakeAt(w: World, x: number, y: number): void {
    const cx = Math.floor(x / CHUNK), cy = Math.floor(y / CHUNK);
    for (let ny = Math.max(0, cy - 1); ny <= Math.min(w.Hc - 1, cy + 1); ny++)
        for (let nx = Math.max(0, cx - 1); nx <= Math.min(w.Wc - 1, cx + 1); nx++)
            w.chunkActive[ny * w.Wc + nx] = WAKE_FRAMES;
}

📚 A Material Library

In the Intermediate tier's engine, materials were a literal array; new ones meant editing that array. Better: a tiny registry — a registerMaterial() builder and a registerReaction() builder. The engine never names a material directly. It only ever asks for properties (.kind, .density, .flammable, .fuel, .color).

Adding a material is a call site, not a code change

Want acid? Call registerMaterial({ name: 'acid', kind: 'liquid', density: 4, color: [120, 200, 80] }) and one registerReaction() for what it eats. The engine never learns the word "acid" — it just sees another liquid in the table and another row in the reaction list.

📝 Click to see: the library API 
// The registry — mutable, populated by registerMaterial calls.
const MAT = [], REACTIONS = [];

function registerMaterial(spec) {
    const id = MAT.length;
    MAT.push(Object.assign({ id }, spec));
    return id;                       // hand back the id for use in reactions
}
function registerReaction(a, b, ra, rb) {
    REACTIONS.push({ a, b, ra, rb });
}

// The standard ten — registered once at module load.
const AIR   = registerMaterial({ name: 'air',   kind: 'air',    color: [13, 17, 30]  });
const WATER = registerMaterial({ name: 'water', kind: 'liquid', density: 3, color: [60, 120, 210] });
// …etc.
registerReaction(WATER, LAVA, STEAM, STONE);

// New material at the call site — no engine edit.
const ACID = registerMaterial({
    name: 'acid', kind: 'liquid', density: 4, color: [120, 200, 80]
});
registerReaction(ACID, STONE, SMOKE, AIR);   // acid dissolves stone
type MaterialSpec = {
    name: string;
    kind: 'air' | 'solid' | 'powder' | 'liquid' | 'gas' | 'fire';
    color: [number, number, number];
    density?: number;
    flammable?: boolean;
    fuel?: number;
};

const MAT: (MaterialSpec & { id: number })[] = [];
const REACTIONS: { a: number; b: number; ra: number; rb: number }[] = [];

function registerMaterial(spec: MaterialSpec): number {
    const id = MAT.length;
    MAT.push(Object.assign({ id }, spec));
    return id;
}
function registerReaction(a: number, b: number, ra: number, rb: number): void {
    REACTIONS.push({ a, b, ra, rb });
}

🎛️ Live Rule Tuning

A material's behaviour is partly the code of its rule, and partly a handful of numbers — how fiercely fire ignites, how viscous lava is, how fast smoke fades. Pulling those numbers into a live CONFIG object lets sliders adjust them in real time.

One source of truth the engine reads each frame

The step function calls into CONFIG.fireIgnite, CONFIG.lavaViscosity, CONFIG.gasDissipate rather than hard-coded constants. Drag a slider, the next frame already reflects it. This is the "painter UI" idea from the sub-track's roadmap — designing rules by feel.

🧪 Mini-Project: a Performant Sandbox

Everything composed. A 75,600-cell world stepped through the chunked engine, with the full ten materials plus Acid (added by the library demo and still present in the registry). FPS readout, awake-chunk count, every interaction emergent.

The whole stack in one glance

  1. Library — materials in the registry, including a runtime-added one.
  2. Engine — per-cell dispatch by kind, moved-flag-safe.
  3. Chunks — only awake chunks step; the rest sleep.
  4. CONFIG — rule parameters live, tunable.
