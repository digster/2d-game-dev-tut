💡 What's different from Terraria voxels: in the Terraria sub-track a tile is ~16px and sits still until the player edits it. Here a cell is one pixel, and every cell runs a rule every frame — the world simulates itself. That changes two things: how you render (you can't fillRect tens of thousands of cells) and how carefully you must think about update order. This tier builds that engine.

🖼️ Pixels, Not Tiles — the ImageData Engine

The world is still a flat Uint8Array of material ids — that storage idea carries over unchanged from Terraria → Tile Data Structures. What changes is scale: a 240×150 grid is 36,000 cells, and redrawing that many with one fillRect each, every frame, is far too slow.

Render through an ImageData buffer

Instead of drawing cells, you write raw RGBA bytes. An ImageData holds a Uint8ClampedArray of 4 bytes per pixel. Write the material colours into it, putImageData it onto a sim-resolution offscreen canvas, then drawImage that offscreen canvas scaled up to the display — with imageSmoothingEnabled = false so the upscale stays crisp and pixelated.

📝 Click to see: the ImageData renderer 
// Build a renderer for a W×H pixel grid, displayed on a larger canvas.
function makeRenderer(canvas, W, H) {
    const ctx = canvas.getContext('2d');
    ctx.imageSmoothingEnabled = false;             // crisp nearest-neighbour upscale

    // An offscreen canvas at the SIMULATION resolution.
    const off = document.createElement('canvas');
    off.width = W; off.height = H;
    const offCtx = off.getContext('2d');
    const img = offCtx.createImageData(W, H);      // W*H*4 bytes, RGBA
    const data = img.data;

    return function render(grid) {
        for (let i = 0; i < grid.length; i++) {
            const p = i << 2;                       // byte offset = i * 4
            const m = grid[i];
            if (m === SAND)      { data[p] = 206; data[p+1] = 180; data[p+2] = 110; }
            else if (m === WALL) { data[p] = 92;  data[p+1] = 96;  data[p+2] = 108; }
            else                 { data[p] = 13;  data[p+1] = 17;  data[p+2] = 30;  }
            data[p + 3] = 255;                     // opaque
        }
        offCtx.putImageData(img, 0, 0);
        // Scale the sim-res image up to fill the display canvas.
        ctx.drawImage(off, 0, 0, W, H, 0, 0, canvas.width, canvas.height);
    };
}

⏳ The Falling-Sand Rule

The whole engine is one rule applied to every sand cell: try to fall. Straight down if you can; otherwise slide down-left or down-right. That's it — pile formation, slopes, the angle of repose all emerge from those three options.

You've met this rule before — at tile scale

The Terraria Intermediate tier ran the same down / down-diagonal rule on 16-pixel tiles, as one mechanic among many. Here it runs on single pixels as the entire world. Same rule, different granularity — and at this granularity the next section's subtlety (update order) suddenly matters a great deal.

📝 Click to see: stepSand 
// Advance every sand cell one step, in place.
function stepSand(grid, W, H, frame) {
    // Sweep direction alternates each frame — see the next section.
    const ltr = (frame & 1) === 0;
    // Bottom-up: visit a cell BEFORE the cell it would move into.
    for (let y = H - 2; y >= 0; y--) {
        for (let k = 0; k < W; k++) {
            const x = ltr ? k : W - 1 - k;
            const i = y * W + x;
            if (grid[i] !== SAND) continue;

            const below = i + W;
            if (grid[below] === AIR) {                 // 1. straight down
                grid[i] = AIR; grid[below] = SAND;
                continue;
            }
            const canL = x > 0     && grid[below - 1] === AIR;
            const canR = x < W - 1 && grid[below + 1] === AIR;
            if (canL || canR) {                        // 2. down-diagonal
                const dir = (canL && canR) ? (Math.random() < 0.5 ? -1 : 1) : (canL ? -1 : 1);
                grid[i] = AIR; grid[below + dir] = SAND;
            }
        }
    }
}

🔀 Update Order & Directional Bias

Here is the subtlety that pixel-scale forces you to confront. The grid is updated in place — each cell reads and writes the same array. So the order you visit cells in changes the result. Get it wrong and sand doesn't fall — it teleports.

The top-down trap

Scan rows top to bottom: you move a grain from row 5 down to row 6 — then the scan continues downward, reaches that same grain at row 6, and moves it again to row 7… and again… The grain falls the entire column in a single frame. No animation, just an instant snap to the floor.

The fix: scan bottom to top. Now a grain is visited before the cell below it, so it moves exactly one row per frame — and you see it fall. A second, subtler bias is horizontal: always sweeping left-to-right nudges piles sideways, so we alternate the sweep direction each frame.

📝 Click to see: the WRONG way (kept to demonstrate the bug) 
// BUGGY ON PURPOSE — a top-down scan. A grain moved down is re-visited
// lower in the SAME pass and moved again, so it teleports to the floor.
function stepSandNaive(grid, W, H) {
    for (let y = 0; y < H - 1; y++) {            // top-down — this is the bug
        for (let x = 0; x < W; x++) {
            const i = y * W + x;
            if (grid[i] !== SAND) continue;
            const below = i + W;
            if (grid[below] === AIR) { grid[i] = AIR; grid[below] = SAND; }
        }
    }
}

See the bug, then fix it

A steady stream of sand pours from the top. Toggle the scan mode: in naive (top-down) the sand vanishes from the source and reappears as a flat layer on the floor — it never falls. Switch to correct (bottom-up) and the falling stream appears.

🎯 Why this matters more later

Bottom-up scanning fixes falling. But the Intermediate tier's water moves sideways and upward too — and no single scan direction is correct for all of them at once. That's when engines reach for double buffering (read from one grid, write to another) — the fully general answer to update order.

🪣 Mini-Project: Bucket of Sand

Everything together: a paint tool for sand and immovable walls, plus erase. Walls need no rule of their own — the sand rule only ever moves a grain into an AIR cell, so a wall is an obstacle for free.

Walls are "sand that never runs"

There is no stepWall. A wall is just a material the falling-sand rule can't move into. Carve a container out of walls, pour sand in, and it settles inside — the bucket "works" without a single line of code about buckets.

📝 Click to see: the disc brush 
// Stamp a filled disc of material into the grid — the painting brush.
function paintDisc(grid, W, H, cx, cy, r, mat) {
    for (let dy = -r; dy <= r; dy++)
        for (let dx = -r; dx <= r; dx++) {
            if (dx * dx + dy * dy > r * r) continue;   // circular, not square
            const x = cx + dx, y = cy + dy;
            if (x < 0 || x >= W || y < 0 || y >= H) continue;
            grid[y * W + x] = mat;
        }
}

// The sand brush only fills AIR (so it doesn't paint over walls); the wall
// and erase brushes overwrite anything.
function paintDisc(grid: Uint8Array, W: number, H: number,
                   cx: number, cy: number, r: number, mat: number): void {
    for (let dy = -r; dy <= r; dy++)
        for (let dx = -r; dx <= r; dx++) {
            if (dx * dx + dy * dy > r * r) continue;
            const x = cx + dx, y = cy + dy;
            if (x < 0 || x >= W || y < 0 || y >= H) continue;
            grid[y * W + x] = mat;
        }
}
