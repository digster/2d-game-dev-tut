🛣️ Beyond predict-and-correct

The Beginner–Advanced track all assumed one shape: authoritative server, clients predict locally, server snapshots correct (now silently, thanks to reconciliation and smoothing). That shape works for most games. But there are two genres where it visibly doesn't, and one scale problem it doesn't touch:

  • Real-time strategy games (Age of Empires, Starcraft, Total Annihilation). You might have hundreds of units on screen. Broadcasting each unit's position to every client every tick blows the bandwidth budget. The fix is lockstep — every machine simulates from the same inputs, so the only thing on the wire is inputs.
  • Fighting games (Street Fighter, Skullgirls, every modern fighter). A single-frame input lag is the difference between a combo working and not. The fix is rollback — predict the opponent's input, run the sim immediately, rewind + re-simulate when the truth arrives.
  • MMOs and big shooters (64+ players, large worlds). You can't send every entity to every player no matter what your encoding does. The fix is area-of-interest filtering — only send the entities near the player, queried via a spatial structure.
💡 All three depend on a property we've assumed since tier 1: DETERMINISM. Lockstep needs every machine to produce the same simulation from the same inputs. Rollback needs every machine to produce the same simulation from the same inputs (the rewinds happen locally). Even AoI needs determinism for replays and reconciliation. We start with how easily floating-point math sabotages it.

🎲 Determinism — why floats betray you

Within a single browser, Math.sin(1.5) gives the same answer every time you call it. Across different machines, that's not guaranteed: different CPUs, different compilers, different vector instruction sets, even different operation orders inside the same compiler optimization level can produce 1-bit differences in the last decimal places. In a one-shot calculation that's invisible. In a long simulation, those 1-bit differences can explode.

Below, a double pendulum — a famously chaotic system. The cyan one starts with θ₂ = π/2 exactly. The red one starts with θ₂ = π/2 + Δθ, where Δθ is the slider (default 1e-7 rad — about one ten-millionth of a degree). Watch the two trails: for the first few seconds they follow each other; then they diverge into completely different shapes.

🤔 But I'm only running one machine — why do I care?

Because every player runs a different machine. Two clients running your lockstep RTS need to agree on every unit's position, every frame, for hours. A 1-bit difference in Math.cos on one client compounds over thousands of operations per frame, billions per session — and within a minute the two clients' worlds have diverged. The game desyncs.

The slider above is a stand-in for that 1-bit cross-machine difference. The chaos you see in the red pendulum is the chaos every long-running float simulation invites.

📝 How shipped lockstep games avoid this — fixed-point math 
// Real lockstep games (StarCraft, Age of Empires II, Total Annihilation,
// every Worms game) use FIXED-POINT MATH for the simulation. A position
// isn't a float — it's an integer scaled by 2^16 (so the integer 65536
// represents 1.0). Every CPU on every platform gets the same answer
// for `int + int`, `int * int`, and `int >> int`. No drift, ever.
//
// The cost: limited range and no exact transcendentals. You build a
// sin/cos lookup table once and use that. You bake constants into
// constants. You give up some accuracy for the bit-exact guarantee.

const Q = 16;                         // 16 fractional bits
function toFixed(f)  { return Math.round(f * (1 << Q)); }
function fromFixed(x){ return x / (1 << Q); }
function fmul(a, b)  { return (a * b) >> Q; }     // (a*b) is exact, then shift
function fdiv(a, b)  { return Math.floor((a << Q) / b); }

// A bit-exact sin via 256-entry lookup. Inputs in fixed-point radians
// scaled by ((1<<Q) * 2π).
const SIN_TABLE = new Int32Array(256);
for (let i = 0; i < 256; i++) {
    SIN_TABLE[i] = toFixed(Math.sin(i / 256 * 2 * Math.PI));
}
function fsin(angle) {
    // angle is fixed-point radians; map to [0..256) then index.
    const idx = ((angle >>> (Q - 8)) & 0xff);  // top 8 bits of fractional rad
    return SIN_TABLE[idx];
}
// This is bit-identical on Intel, ARM, RISC-V, anywhere JS or your
// compiled language runs. Two machines that compute the same sequence
// of fmul/fadd/fsin calls will agree to the last bit.
const Q: number = 16;
function toFixed(f: number): number   { return Math.round(f * (1 << Q)); }
function fromFixed(x: number): number { return x / (1 << Q); }
function fmul(a: number, b: number): number { return (a * b) >> Q; }
function fdiv(a: number, b: number): number { return Math.floor((a << Q) / b); }

const SIN_TABLE = new Int32Array(256);
for (let i = 0; i < 256; i++) {
    SIN_TABLE[i] = toFixed(Math.sin(i / 256 * 2 * Math.PI));
}
function fsin(angle: number): number {
    const idx = ((angle >>> (Q - 8)) & 0xff);
    return SIN_TABLE[idx];
}
💡 In JS specifically: Number is a 64-bit float by design. Real fixed-point work usually happens in a separate physics engine layer or in WebAssembly, where you can pick the integer width. For pedagogical demos the lookup-table fixed-point pattern above is enough to feel correct.

🔒 Lockstep — bandwidth = O(players)

If every machine runs the same deterministic simulation, you don't need to broadcast unit positions. You just need to broadcast inputs. Each player sends their commands once per tick; the server (or every peer in a P2P lockstep game) collects everyone's inputs and re-broadcasts them; every machine simulates forward from the same state, with the same inputs, and produces the same world.

The bandwidth math is shocking: a 200-unit RTS at 20 Hz costs about ~64 kbps per client if you snapshot all positions, but only ~0.5 kbps per client if you lockstep the inputs. Drag the units slider below — the gap grows linearly.

🛑 The lockstep tax

Lockstep's win is bandwidth. Its cost is everyone advances at the speed of the slowest peer. A single laggy player makes the whole game stutter. Hidden-from-the-player techniques like input delay (always run ~50 ms behind to give every peer time to send their input) hide a lot of it, but lockstep RTS games still struggle on bad connections. Modern variants (rollback-on-top-of-lockstep, "stop-and-go" recovery) push the limits.

⏪⏩ Rollback netcode — predict and rewind

Lockstep waits for the slowest peer. Fighting games can't afford that: a single frame of input lag is a competitively-meaningful delay. Rollback netcode (popularised by GGPO in 2006, now the default for every shipped fighter) flips lockstep on its head:

  1. Every tick, the client PREDICTS the opponent's input (= same as their last known input).
  2. Run the simulation immediately, locally.
  3. Save the world state in a ring buffer.
  4. When the opponent's REAL input for tick T arrives, compare to the prediction.
  5. If different: restore world state to tick T, fix the input, re-apply every tick from T forward.

📡 Area of interest — spatial filtering

A 64-player battle royale with five projectiles each = 320 entities. At 20 Hz with 16 bytes per entity that's ~819 kbps per client for everyone's state. You can't pay that on mobile. The fix: only send the entities near the player. Area-of-interest (AoI) filtering, also called "relevancy" or "interest management", queries a spatial structure for the entities within a radius of the observer and includes only those in each snapshot.

Below, 100 entities wander a big world. Move your mouse over the panel — your cursor is the observer. The AoI radius is the cyan circle. Entities inside it are green (sent in the snapshot); entities outside are dim (omitted). The faint green rectangles are the spatial-grid cells the query touched — at most 4 cells, since the cell size is set equal to the AoI radius.

🛡️ Anti-cheat primer

Once you're shipping over the open Internet, players will try to cheat. This isn't a malicious-attacker-only concern — even well-meaning players will run modded clients, share automation, or accidentally exploit a state-machine bug. A complete anti-cheat story is its own field (kernel-level drivers, server-side machine-learning, third-party services like BattlEye and Easy Anti-Cheat), but every game needs the same baseline:

🔒 Server authority — the foundation

The server is the source of truth. The client can request actions (move here, shoot there); the server decides whether they're legal. Never trust the client's reported damage, health, or position — they're inputs to a validation, not outputs. Authoritative server (the architecture you've been using all track) is necessary but not sufficient.

📏 Sanity bounds — the second line

Every input has a physical bound. Velocity can't exceed MAX_SPEED; a player can't fire faster than FIRE_INTERVAL; a unit can't appear in a tile they don't have line-of-sight to. Check these on the server. A speed hack that teleports the player 1000 px/frame is immediately detectable; the sneakier "always 10% faster than allowed" cheat needs statistical detection (averaging speed over many frames + flagging anomalies).

🎬 Replay verification — the audit

A determinism-based architecture (lockstep, rollback) gets free cheat detection: record everyone's inputs to the match. After the match, replay them on a trusted server. If the result differs from what the players reported, someone cheated. Shipped fighters (e.g. Street Fighter 6) and shipped RTS (e.g. SC2) both use this to award bans long after the match is over.

💡 The single most important rule: if a cheater can do it with a modified game client, the architecture is broken. Every fix has to be on the server side. A client-side anti-cheat is at best a speed bump (since the cheater controls the client). The Beginner-tier discipline of "server-state and client-state are SEPARATE objects" is what enables every later anti-cheat technique to exist.

🧰 Recap — what you've got

  • Determinism — the property every cross-machine technique depends on, and the reason real lockstep games use fixed-point math instead of floats.
  • Lockstep — bandwidth = O(players × inputs/tick) regardless of world size. The RTS architecture. Costs: everyone waits for the slowest peer, requires perfect determinism.
  • Rollback netcode — predict opponent input, rewind + re-simulate when wrong. The fighting-game architecture. Costs: requires perfect determinism + a state ring buffer + frequent re-simulation under prediction load.
  • Area-of-interest filtering — only send entities near the observer. The scale architecture (MMOs, big shooters, sandbox survival). Pairs with the Fundamentals tier's spatial structures.
  • Anti-cheat baseline — server authority + sanity bounds + replay verification. Every shipped multiplayer game needs at least these three.
