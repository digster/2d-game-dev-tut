👑 Authority — who has the truth?

Every online game has to answer one question before any other: when two machines disagree about the state of the world, who wins? The answer shapes everything about the architecture.

Model Who decides truth? Used by Trade-off
Authoritative server One dedicated server. Clients send inputs; the server sends back state. Most shooters, MMOs, MOBAs Cheat-resistant; costs infrastructure; client always feels remote.
Listen server One of the players' machines IS the server. Many co-op games, casual shooters Zero infrastructure; host has unfair latency advantage; trivially cheatable.
P2P with input lockstep No server; every client runs the same deterministic sim from the same inputs. RTS (Age of Empires-era), some fighters Tiny bandwidth (just inputs!); requires perfect determinism; one cheater desyncs the match.
P2P with rollback Predict opponent input; rewind and re-simulate when the real input arrives. Fighting games (GGPO-era) Feels instant locally; requires perfect determinism + frequent re-simulation.
💡 This tier uses authoritative-server everywhere because it's the most common shape and because the technique vocabulary you learn here (prediction, interpolation, smoothing) is the foundation that the Expert tier's lockstep and rollback work later builds on. Even if your eventual target is lockstep RTS, you still need this tier — the client-rendering smoothing techniques are universal.

The two-clock setup

Every demo in this tier runs both sides — server and client — in the SAME browser tab, communicating only through a FakeNetwork object. That's the only way to teach this without a real backend. Discipline: never read server state directly from a client renderer. If a demo does, it's lying about what the client actually knows.

The visual convention: every demo has a SERVER panel on the left (orange) and a CLIENT panel on the right (cyan). The two panels show the SAME world from two different perspectives. Watching the gap between them IS the whole pedagogy of the tier.

🔮 Client-side prediction — apply input now

The instant the player presses a key, the client applies the input locally AND sends it to the server. The screen responds immediately; the server's later acknowledgement is just a sanity check. When the server's snapshot arrives, we naively snap the client to the authoritative position.

Toggle prediction OFF below to compare directly: with prediction, motion is instant; without it, you're back to the naive demo's RTT lag. The green dashed ghost on the CLIENT panel shows the authoritative position from the last snapshot — the gap between the cyan ball and the green ghost is the disagreement that produces visible snaps.

🔍 What you should see

  1. Prediction OFF: the CLIENT ball lags behind the SERVER ball by exactly RTT. Identical to the naive demo.
  2. Prediction ON, low jitter: the CLIENT ball matches the SERVER ball almost perfectly. The green ghost overlay sits right on top of the cyan ball — they agree.
  3. Prediction ON, high jitter (push to 80+): the ghost separates from the cyan ball occasionally and the cyan ball "snaps" to it on every snapshot. Visible micro-stutter.

Snapping on every disagreement is the BAD version of reconciliation. The Advanced tier introduces server reconciliation — keep a ring buffer of unacked inputs, apply the server snapshot, then replay your buffered inputs on top. The visible snap disappears because the prediction realigns with the server's view of the past.

🪞 Entity interpolation — render remotes in the past

Prediction fixes the LOCAL player. But what about every OTHER player — the ones whose inputs you don't have? You can't predict them; all you have is the periodic snapshots the server sends. If you just render the latest snapshot directly, remote players stutter at the snapshot rate. The fix: render remotes ~one snapshot-interval IN THE PAST, lerping between the two snapshots that surround the chosen render time.

The remote entity below moves smoothly in a circle on the server. The two panels show the same data, rendered two different ways:

  • Snap to latest (purple) — render whatever snapshot arrived most recently. Stutters at low snapshot rates.
  • Interpolation buffer (green) — render at now − interpDelay, lerping between the two snapshots straddling that time.

🔍 What to try

  1. Start at 5 Hz snapshot rate. The LEFT (snap-to-latest) ball teleports five times per second — visibly stuttery. The RIGHT (interp) ball glides smoothly.
  2. Watch the orange dashed outline on the RIGHT panel — that's where the entity ACTUALLY is on the server right now. The smooth green ball is visibly behind it. That's the cost of the interp buffer.
  3. Drop snapshot rate to 2 Hz. The LEFT ball jumps every half-second. The RIGHT ball still glides but now sits ~500 ms behind reality. Hit-detection on remotes at this rate needs the Advanced tier's lag-compensation.
  4. Push snapshot rate to 30 Hz. Both balls now look identical — the buffer's cost is hidden at high rates.
💡 The seed planted in Beginner: remember the tick-vs-frame demo's "interpolate between ticks" toggle? This is the same technique, but the "ticks" you're lerping between are now snapshots arriving from another machine instead of locally-simulated steps. Same idea, different data source.

🪶 Snap vs smooth — softening the correction

Even with prediction working perfectly, the server will occasionally disagree with the client. When that happens, do you snap the client to the authoritative position instantly, or do you slide there over a small smoothing window?

Below, both players move in a straight line; every ~1.5 seconds a "correction event" teleports them to a new Y. The LEFT player snaps instantly; the RIGHT player smooths the correction with frame-rate-correct exponential decay (99% of the gap closed in smoothMs).

📝 Click to see: framerate-correct exponential smoothing in code 
// Smooth the player position toward `target` over ~smoothMs milliseconds.
// Frame-rate INDEPENDENT — same visible smoothness at 30 fps or 240 fps.
//
// Key idea: lerp(current, target, 1 - exp(-k * dt)) where k is chosen so
// that ~99% of the remaining gap closes within smoothMs. That gives the
// exponential-decay "drag toward the target" feel that hides corrections
// without ever overshooting.

function smoothToward(currentPos, targetPos, smoothMs, dt) {
    if (smoothMs <= 0) return { ...targetPos };          // 0 ms = snap
    const k = -Math.log(0.01) / (smoothMs / 1000);         // 1/s
    const alpha = 1 - Math.exp(-k * dt);
    return {
        x: lerp(currentPos.x, targetPos.x, alpha),
        y: lerp(currentPos.y, targetPos.y, alpha),
    };
}

// The naive version `lerp(current, target, smoothness * dt)` is what people
// reach for first — but it's WRONG. At high dt (a tab refocus, a slow frame),
// `smoothness * dt` can exceed 1, the lerp overshoots, the player snaps
// anyway. Exponential decay is bounded — alpha is always in [0, 1).
type Vec2 = { x: number; y: number };

function smoothToward(currentPos: Vec2, targetPos: Vec2, smoothMs: number, dt: number): Vec2 {
    if (smoothMs <= 0) return { ...targetPos };
    const k = -Math.log(0.01) / (smoothMs / 1000);
    const alpha = 1 - Math.exp(-k * dt);
    return {
        x: lerp(currentPos.x, targetPos.x, alpha),
        y: lerp(currentPos.y, targetPos.y, alpha),
    };
}
💡 Why 1 - exp(-k·dt): this formula is the standard way to make any "drag toward a target" behaviour frame-rate-independent. It comes up everywhere in game code — follow cameras, audio fades, network corrections. Racing-sim's expert tier uses the same trick for the follow camera; the technique transfers cleanly between domains.

🧰 Recap — what you've got

  • A vocabulary for authority models — authoritative server, listen server, P2P lockstep, P2P rollback — and a working mental model for why authoritative server is the most common default.
  • Client-side prediction — apply local input immediately, send to server, snap to authoritative on snapshot arrival. The single biggest UX win in any networked game.
  • Entity interpolation — render REMOTE players ~one-snapshot-interval in the past so you can lerp between two known snapshots. Smooths remotes at low snapshot rates at the cost of a small render-time deficit.
  • Frame-rate-correct smoothing via 1 − exp(−k·dt) — hides correction snaps without overshooting. Same formula transfers to follow cameras, audio fades, anywhere a "drag toward target" feel is wanted.
  • The capstone arena demo, where all three techniques compose: local player feels instant, remote feels smooth, corrections disappear into smoothing.
💡 What you DON'T have yet (the Advanced-tier promise): you're still snapping on prediction-vs-authoritative disagreement. Server reconciliation fixes that: keep a ring buffer of every input you sent, and when the server says "as of tick N your state was X", apply X and then REPLAY all your buffered inputs from N+1 onward. The visible snap disappears — your prediction realigns with the server's view of the past. You also haven't done anything about WHO can be sure where they hit at high latency: that's lag compensation, the Advanced tier's other big technique.
