🎯 From "playable" to "polished"

The Intermediate tier got you to a game that worked over a bad network: local players felt instant via client-side prediction, remote players moved smoothly via entity interpolation, and corrections were softened via exponential smoothing. But two visible artefacts and one invisible cost still remain:

  • The prediction snap — every server snapshot teleports your client to the authoritative position. We hid it under smoothing, but it's still there.
  • The "behind reality" deficit for remote players — you aim at where the target was, not where the server thinks it is right now. Your shots miss things you visibly hit.
  • Bandwidth blows up with player count. The Beginner-tier bandwidth equation said it would; once you have 16 entities at 30 Hz with float positions, you're a Mbps per client.

This tier fixes all three.

💡 The shape of the techniques: reconciliation and lag-comp are extensions of authority — both have the server use HISTORICAL state rather than current state in specific situations. Delta and quantization are encoding wins — they shrink each message without changing what's communicated. Together, they're the four techniques that turn a working game into a shippable one.

🔁 Server reconciliation — replay buffered inputs

Client-side prediction (Intermediate) was applying input locally and snapping to the server's snapshot when one arrived. The snap was visible because, by the time the snapshot reached the client, the client had already moved further forward in prediction-land than the snapshot reflected.

Reconciliation fixes that with three small changes:

  1. Tag every input message with a monotonic tick. The client also keeps a buffer of the inputs it sent.
  2. Server snapshots carry an "ack tick" — the highest input tick the server has processed.
  3. On snapshot arrival, the client drops acked inputs from the buffer, snaps to authoritative, then replays the remaining buffered inputs.

Result: the client realigns with the server's view of the PAST plus its own predicted-but-not-yet-acked motion. Because the predicted past matched truth, the replay leaves the present unchanged — no visible snap.

// CLIENT — on every frame, predict locally and buffer the input.
let nextTick = 1;
const inputBuffer = [];   // {tick, ax, ay, dt}

function clientFrame(dt, input) {
    // 1. Predict — apply input + integrate against local state.
    applyInput(clientPlayer, input);
    integrate(clientPlayer, dt);

    // 2. Send input to server, tagged with a tick we control.
    const tick = nextTick++;
    inputBuffer.push({ tick, ax: input.ax, ay: input.ay, dt });
    sendToServer({ kind: 'input', tick, ax: input.ax, ay: input.ay });
}

// CLIENT — on snapshot arrival.
function onSnapshot(snap, ackTick) {
    // 3. Drop acked inputs — the server has already accounted for them.
    while (inputBuffer.length > 0 && inputBuffer[0].tick <= ackTick) {
        inputBuffer.shift();
    }

    // 4. Snap to authoritative.
    clientPlayer.x = snap.x; clientPlayer.y = snap.y;
    clientPlayer.vx = snap.vx; clientPlayer.vy = snap.vy;

    // 5. Replay every buffered input on top, in order.
    //    This is "reconciliation" — your prediction realigns with the
    //    server's view of the past, then carries forward your unacked motion.
    for (const b of inputBuffer) {
        applyInput(clientPlayer, { ax: b.ax, ay: b.ay });
        integrate(clientPlayer, b.dt);
    }
}

// SERVER — track the highest input tick ever processed.
let highestAck = 0;
function onClientInput(msg) {
    applyInput(serverPlayer, msg);
    if (msg.tick > highestAck) highestAck = msg.tick;
}
// And ship it with every snapshot:
function sendSnapshot() {
    sendToClient({ kind: 'snapshot', snap: snapshot(serverPlayer), ackTick: highestAck });
}
type InputBufEntry = { tick: number; ax: number; ay: number; dt: number };
type Snapshot       = { x: number; y: number; vx: number; vy: number };
type ServerMsg      = { kind: 'snapshot'; snap: Snapshot; ackTick: number };

let nextTick = 1;
const inputBuffer: InputBufEntry[] = [];

function clientFrame(dt: number, input: { ax: number; ay: number }): void {
    applyInput(clientPlayer, input);
    integrate(clientPlayer, dt);
    const tick = nextTick++;
    inputBuffer.push({ tick, ax: input.ax, ay: input.ay, dt });
    sendToServer({ kind: 'input', tick, ax: input.ax, ay: input.ay });
}

function onSnapshot(snap: Snapshot, ackTick: number): void {
    while (inputBuffer.length > 0 && inputBuffer[0].tick <= ackTick) {
        inputBuffer.shift();
    }
    Object.assign(clientPlayer, snap);
    for (const b of inputBuffer) {
        applyInput(clientPlayer, b);
        integrate(clientPlayer, b.dt);
    }
}
💡 The buffer is your future. Once you start reconciling, the in-flight inputs are simply "what will happen next" from the server's perspective. You can pre-render moves before the server confirms them, and the moment authoritative truth arrives, your prediction snaps invisibly into place instead of jumping.

⏪ Lag compensation — hit-check in the past

The local player feels instant (prediction + reconciliation). Remote players move smoothly but visibly behind reality (entity interpolation). When the local player shoots a remote, they aim at where they see the remote — which is in the past. By the time the fire message reaches the server, the target has moved. The shot misses the current position, even though it would have hit the position the player saw.

Lag compensation fixes this on the server side. The server keeps a ring buffer of past world states. The fire message carries the client's believed render-time. The server rewinds to that time, checks the shot, and tells the client whether they hit. The "they aimed at what they saw" promise is kept.

Below, both panels run the same target on the same orbit. Auto-fire shoots at the target every second. The LEFT panel has no lag-comp — the server validates shots against the target's current position. The RIGHT panel has lag-comp on — the server rewinds to the client's render-time. The faint orange dots on the right panel are the server's stored history (= the rewind buffer).

🔍 What to try

  1. Crank target speed to 3 rad/s and RTT to 400 ms. The LEFT (no-comp) hit rate plummets — the target moves visibly faster than your shot can chase. The RIGHT (lag-comp) hit rate stays near 100%.
  2. The visible "miss" cost on the left is the price of authority being on the wrong side of latency. Lag-comp doesn't change the architecture; it changes when the authority looks.
💡 The cost of lag-comp: the shooter (you) gets fair hits, but the shootee (the remote you hit) sometimes "dies behind cover" — the server validated a shot at where they used to be. There's no free lunch in netcode; lag-comp picks the shooter's side of the fairness trade. Most shipped shooters accept this; some (e.g. Apex Legends, Valorant) tune the rewind window heavily to keep it bounded.

📉 Delta compression — only send what changed

The Beginner-tier bandwidth equation said bytes/sec = tickRate × playerCount × entitiesPerPlayer × bytesPerEntity. That assumes you send every entity, every tick. Delta compression says: only send entities whose state actually changed since the last snapshot the client acknowledged. In a world where most entities are stationary most of the time, that's a 5–100× win.

  2. Drop entities to 8, change rate to 80% (a tight arena shooter). Delta is only marginally smaller — most things move.

Real implementations also encode whether a field within an entity changed (sub-entity deltas), and combine that with bit-packing for further gains. Quantization (below) is what makes the "bytes per entity" number small in the first place.

📦 Quantization — float → int16

A 32-bit float can express any position with ~7 decimal digits of precision. Your player probably can't see the difference between x=123.0001234 and x=123. Quantization trades that invisible precision for fewer bytes: map a float in a known range (say ±512 world units) into a k-bit integer, send that, and decode on the other side.

// Map `value` ∈ [-range, +range] to a `bits`-wide signed integer.
function quantize(value, range, bits) {
    const max = (1 << (bits - 1)) - 1;        // e.g. bits=16 → max=32767
    const clamped = Math.max(-range, Math.min(range, value));
    return Math.round(clamped / range * max);
}

// Reverse — back to the corresponding float in [-range, +range].
function dequantize(q, range, bits) {
    const max = (1 << (bits - 1)) - 1;
    return q / max * range;
}

// Worst-case error per dimension is one quantization step.
// For ±range and bits-wide ints:  step = range / ((1 << (bits-1)) - 1)
// A 1024×1024 world at 16 bits ≈ 0.03 world-units of error — invisible.
function quantize(value: number, range: number, bits: number): number {
    const max = (1 << (bits - 1)) - 1;
    const clamped = Math.max(-range, Math.min(range, value));
    return Math.round(clamped / range * max);
}
function dequantize(q: number, range: number, bits: number): number {
    const max = (1 << (bits - 1)) - 1;
    return q / max * range;
}
💡 Bit-packing is the next step. Once you've quantized your fields to small integers, you can pack multiple fields into one machine word instead of sending them as separate JSON values. Real libraries (e.g. flatbuffers, capnproto, or hand-rolled bitstreams) push this aggressively — an entity update that was 48 bytes of JSON can shrink to 6–8 bytes of packed bits. The Fundamentals tier's network section hints at this; production shooters live by it.

🧰 Recap — what you've got

  • Server reconciliation — the input buffer + replay loop that hides every prediction snap. The Intermediate tier's predictionDemo with smoothing was hiding the snap; this tier ELIMINATES it.
  • Lag compensation — server-side history rewind so hit-checks happen at the client's render-time. Shooters fire fairly; the trade-off is that the shootee may "die behind cover".
  • Delta compression — encode only what changed since the last ack. 5–100× wins on idle-heavy worlds; small wins on tight shooter arenas.
  • Quantization — map float fields into k-bit integers over a known range. Halves to quarters your per-entity wire size for invisible loss of precision.
  • A capstone 2-player shooter where the four techniques compose under harsh network sliders.
💡 What you DON'T have yet (the Expert-tier promise): you're still running a "client predicts, server is authority" model. For games that need PERFECT determinism across machines (RTS, fighters), there are two harder architectures: lockstep (everyone runs the same sim from the same inputs — used by Age of Empires; bandwidth = O(players), not O(units)) and rollback netcode (predict opponent inputs, rewind + re-simulate when the truth arrives — used by every modern fighter). You also haven't addressed area-of-interest — at 64+ players you can't send everyone's state to everyone. All four land in Expert.
