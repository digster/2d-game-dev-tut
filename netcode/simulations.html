⚖️ Lockstep vs rollback, side by side

The two determinism architectures from the Expert tier, running the same scenario so you can compare them directly. A remote target jumps around; both panels track it, with the target's "input" arriving over a FakeNetwork at the RTT you set.

  • LOCKSTEP (green) can't advance a tick until that tick's remote input has arrived. The whole scene runs RTT/2 in the past — smooth and consistent, but laggy.
  • ROLLBACK (yellow) predicts the remote input and advances immediately, correcting (snapping) when the prediction was wrong. Always instant, occasionally jumpy.
starting up…
💡 The trade in one sentence: lockstep buys consistency at the cost of input delay; rollback buys zero input delay at the cost of occasional visual corrections. Crank RTT up — lockstep's input delay grows linearly (the dot lags further behind the orange truth), while rollback's correction count grows instead (the dot snaps more often but never lags). Fighting games chose rollback because a player tolerates an occasional snap far better than constant input lag.

🔥 Area-of-interest heatmap

The Expert tier showed AoI filtering from one observer's point of view. At server scale, what matters is the aggregate: how many players are interested in each region of the world. This heatmap colours each grid cell by how many players' AoI circles overlap it. Hot (red) cells are expensive — the server must replicate their entities to many clients at once.

🔍 What to try

  1. Slide clustering to 100%. All players pile into the centre — one blazing-hot region the server must work hardest to serve. This is why "everyone fights over the same objective" is a server's worst case.
  2. Drop clustering to 0%. Players spread out, the heat evens out, peak cell interest drops. Spreading players is itself an interest-management strategy (sharding, zoning).
  3. Cross-link: the grid here is the same spatial structure from Spatial Partitioning & QuadTree in the Fundamentals tier, now used to measure load rather than to query.

⏺️ Replay scrubber — the determinism payoff

The whole track rests on one assumption introduced in the very first scaffold commit: a deterministic simulation plus a recorded seed (and inputs) can reproduce a session exactly. This demo makes that tangible. Six balls bounce under a seeded, fixed-step simulation; every tick's full state is recorded. Pause and scrub the timeline. Then hit Verify replay: the demo re-runs the sim from scratch using only the seed and the tick count, and asserts the reconstruction is bit-identical to what was recorded.

starting up…
💡 Why this is the capstone: "record seed + inputs, replay bit-identically" is the single mechanism behind three separate techniques you met this track — rollback netcode (re-simulate the recent past), lockstep (every machine replays the same inputs), and anti-cheat replay verification (re-run the match on a trusted server and compare). They're all the same trick, and that trick only works because the simulation is deterministic. The green "bit-identical" message is the foundation the entire track stands on, proven in your browser.
📝 Click to see: the record-and-verify pattern in code 
// A deterministic sim: state is a pure function of (seed, tickCount).
// The ONLY randomness is the seeded initial conditions; after that it's
// pure integration. So recording the seed + counting ticks is enough to
// reconstruct ANY frame — that's what rollback, lockstep, and anti-cheat
// replay all rely on.

const SEED = 777;
function initBalls(seed) {
    const rng = new SeededRng(seed);          // same PRNG since the scaffold
    return Array.from({ length: 6 }, () => ({
        x: rng.range(30, W - 30), y: rng.range(30, H - 30),
        vx: rng.range(-90, 90),   vy: rng.range(-90, 90),
    }));
}
function step(balls) {
    for (const b of balls) {
        b.x += b.vx * DT; b.y += b.vy * DT;
        if (b.x < 8 || b.x > W - 8) b.vx = -b.vx;   // bounce
        if (b.y < 8 || b.y > H - 8) b.vy = -b.vy;
    }
}

// VERIFY: re-run from the seed for N ticks, compare to the recorded frame.
function verify(history) {
    const targetTick = history.length - 1;
    const replay = initBalls(SEED);
    for (let t = 0; t < targetTick; t++) step(replay);
    // Compare replay to history[targetTick] — must be bit-identical.
    const recorded = history[targetTick];
    const maxErr = Math.max(...replay.map((b, i) =>
        Math.max(Math.abs(b.x - recorded[i].x), Math.abs(b.y - recorded[i].y))));
    return maxErr === 0;   // true on any deterministic sim
}
type Ball = { x: number; y: number; vx: number; vy: number };

const SEED = 777;
function initBalls(seed: number): Ball[] {
    const rng = new SeededRng(seed);
    return Array.from({ length: 6 }, () => ({
        x: rng.range(30, W - 30), y: rng.range(30, H - 30),
        vx: rng.range(-90, 90),   vy: rng.range(-90, 90),
    }));
}
function step(balls: Ball[]): void {
    for (const b of balls) {
        b.x += b.vx * DT; b.y += b.vy * DT;
        if (b.x < 8 || b.x > W - 8) b.vx = -b.vx;
        if (b.y < 8 || b.y > H - 8) b.vy = -b.vy;
    }
}
function verify(history: Ball[][]): boolean {
    const targetTick = history.length - 1;
    const replay = initBalls(SEED);
    for (let t = 0; t < targetTick; t++) step(replay);
    const recorded = history[targetTick];
    const maxErr = Math.max(...replay.map((b, i) =>
        Math.max(Math.abs(b.x - recorded[i].x), Math.abs(b.y - recorded[i].y))));
    return maxErr === 0;
}
