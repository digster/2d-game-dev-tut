🛰️ What netcode actually is

Netcode is the bag of techniques that makes a game playable when one player presses a button in Tokyo and another player needs to see the result in Berlin. That sentence hides three brutal physical realities:

  • The speed of light is finite. The fastest possible Tokyo–Berlin round trip is roughly 90 ms, ignoring routers, switches, fibre splices, and the fact that the great-circle path is unlikely to be available.
  • The internet isn't a wire — it's a queue of queues. Every router between you and your peer can hold your packet, reorder it, drop it, or briefly hold it for someone else's flow.
  • You don't have your opponent's state — you have an old copy of it. By the time you see them shoot, they've already moved.
💡 The whole netcode track is a response to those three sentences. Every technique in every later tier — prediction, interpolation, reconciliation, lag compensation, rollback, lockstep — is a way of pretending those problems don't exist while making the game feel like they don't.

The zero-backend setup

Every demo in this track runs inside one browser tab. There is no server. There is no npm install. Instead, every message goes through a simulated network — a class called FakeNetwork with sliders for round-trip time, jitter, packet loss, and reorder probability. You can drag the sliders to nightmare conditions and watch each technique earn (or fail to earn) its keep.

🧱 Two helpers, loaded on every page in this track

  • SeededRng — a deterministic random-number generator. Same seed + same actions = same outcome, every time. Crucial for "see, that's the exact moment the client mispredicted".
  • FakeNetwork — the message-queue scheduler with the four sliders. Endpoint surface (send(targetId, msg) + onMessage(handler)) is the same shape you'd see talking to a WebSocket, so the demo code would survive a future swap to a real transport.

Both already shipped in the scaffold commit. Open the track index and scroll to the bottom — the Scaffold Self-Check panel proves both helpers loaded and behave.

📐 Anatomy of a network message — the four numbers that decide your fate

When you call endpoint.send(targetId, msg) on a real or simulated network, four things determine whether — and when — the recipient will see it:

Term In plain language Realistic range
RTT (round-trip time) Time for a packet to go to the server AND come back. One-way latency is half of that. 20–80 ms local · 80–150 ms regional · 200–400 ms intercontinental · 600 ms+ mobile-on-a-bad-day
Jitter Variance in latency. A 100 ms ± 30 ms link means packets arrive anywhere between 70 ms and 130 ms after sending. 2–10 ms cable · 20–50 ms wifi · 30–100 ms mobile
Packet loss Fraction of packets that simply never arrive. On UDP (which most games use) there is no automatic retry — your code has to deal. < 0.5% good · 1–3% noticeable · > 5% the game starts breaking
Reorder Packet #5 arrives before packet #4 because they took different routes through the network. On UDP, your receiver has to sort them out. Rare on a single link, common across the open Internet

🤔 Why isn't packet loss == lag?

On TCP (browsers, HTTP, anything reliable) lost packets ARE retransmitted, so loss shows up as a sudden lag spike for the duration of one or two retransmits. On UDP (almost all multiplayer games) lost packets just vanish — the receiver gets nothing for that send. Your code has to either resend (input messages) or skip-and-extrapolate (state snapshots). The Intermediate, Advanced, and Expert tiers spend a lot of time on which response is right when.

The FakeNetwork API in three lines

📝 Click to see: the minimum FakeNetwork call shape 
// Build one network with realistic Tokyo-Berlin numbers.
const net = new FakeNetwork({ rttMs: 280, jitterMs: 40, lossRate: 0.01, reorderRate: 0.02 });

// Each player or service registers a name; the call returns an endpoint object.
const client = net.connect('client');
const server = net.connect('server');

// One side listens for messages addressed to its id.
server.onMessage((from, msg) => {
    console.log('server got', msg, 'from', from);
});

// The other side sends. The message will arrive ~140 ms later (one-way = rtt/2)
// PLUS up to ±40 ms jitter, OR not at all (1% loss), OR way later (2% reorder).
client.send('server', { kind: 'input', tick: 12, ax: 1 });

// Every frame: advance the network's internal clock so anything whose
// delivery time has elapsed gets handed to its target endpoint.
function frame(now) {
    net.tick(now);
    requestAnimationFrame(frame);
}
requestAnimationFrame(frame);
// FakeNetwork ships its types on `window`; we mirror them here for clarity.
type NetParams = { rttMs?: number; jitterMs?: number; lossRate?: number; reorderRate?: number; seed?: number };
type Endpoint = {
    id: string;
    send(targetId: string, msg: unknown): void;
    onMessage(handler: (from: string, msg: unknown) => void): void;
};

declare const FakeNetwork: new (opts?: NetParams) => {
    connect(id: string): Endpoint;
    tick(nowMs: number): void;
    setParams(opts: NetParams): void;
    stats: { sent: number; delivered: number; dropped: number; reordered: number };
};

const net = new FakeNetwork({ rttMs: 280, jitterMs: 40, lossRate: 0.01, reorderRate: 0.02 });
const client: Endpoint = net.connect('client');
const server: Endpoint = net.connect('server');

server.onMessage((from, msg) => { console.log('server got', msg, 'from', from); });
client.send('server', { kind: 'input', tick: 12, ax: 1 });

function frame(now: number): void { net.tick(now); requestAnimationFrame(frame); }
requestAnimationFrame(frame);
💡 The endpoint surface is intentionally WebSocket-shaped. The same three calls (connect for the lifecycle, send for outbound, onMessage for inbound) are what you'd use against a real socket later — so this teaching code would survive a swap to a real transport with minimal changes.

🎢 See it — the packet lane

A single-direction lane from CLIENT to SERVER. Auto-sends one packet every 700 ms. Each in-flight packet animates from sent to delivered based on its assigned delay. Drag the sliders to see what each parameter actually does:

  • RTT stretches or compresses the lane animation (lane speed = one-way = rtt/2).
  • Jitter makes the once-regular cadence of arrivals lumpy.
  • Loss shows packets vanish (faded red ✕ above the lane).
  • Reorder tints occasional packets purple and gives them a halo — these will arrive AFTER ones that were sent later.

⏱️ Tick rate vs frame rate

Every multiplayer game has TWO clocks:

  • Tick rate — how often the SIMULATION steps. 20 Hz, 30 Hz, 60 Hz, sometimes higher for fighting games. This is the rate the server makes decisions at, and the rate the client sends inputs at.
  • Frame rate — how often the screen redraws. Driven by requestAnimationFrame in the browser, capped by the monitor refresh (usually 60 Hz, sometimes 144).

The two clocks aren't aligned — three frames can fall between two ticks at 20 Hz / 60 fps, and that gap is where rendering choices matter. Draw the latest tick state directly and the world stutters in chunks; interpolate between the previous tick and the current one and motion stays smooth.

starting up…

🔍 What to try

  1. Start at tick rate 10 Hz with interpolation ON. Motion looks smooth even though the SIM only updates every 100 ms.
  2. Turn interpolation OFF. The orange SIM ball still moves smoothly (you see it on every frame), but if your eye watches the CYAN REN ball it now snaps to the new tick position instead of sliding.
  3. Slide tick rate up to 60 Hz. The two balls now look almost identical — high tick rate hides the interpolation question. That's why low-tick MMOs care about interpolation more than high-tick fighters.
💡 Why this matters for the Intermediate tier: entity interpolation (rendering remote players ~100 ms in the past so you can lerp between two received snapshots) is the same technique you just toggled here, just running on data coming from another machine instead of a local simulation. That's why this demo lives in Beginner — it's the seed the Intermediate tier pays off.

📊 The bandwidth tradeoff equation

Per-client outbound bandwidth (what the server has to push to one player) follows one equation, the equation every netcode budget conversation eventually traces back to:

bytes/sec = tickRate × playerCount × entitiesPerPlayer × bytesPerEntity

Server-total upstream is that × playerCount (the server pushes to every player). Drag the sliders to see how the four knobs combine. Hit a preset to load realistic numbers from shipped games:

🧮 Where the cost actually lives

Notice how brutally players × entities/player scales. A 64-player battle royale tracking just the players themselves is borderline acceptable; track 5 projectiles each and you've doubled your bandwidth. The Expert tier's area-of-interest work shrinks the effective playerCount for each client to "the ones near me", and the Advanced tier's delta compression + quantization shrinks bytesPerEntity by 3–10×. The Beginner-tier equation is the budget; later tiers earn back the budget by attacking each multiplier.

  • A mental model of tick rate vs frame rate, and the fact that the gap between them is where rendering smoothness lives.
  • The bandwidth tradeoff equation — every later tier earns back budget by attacking one of its multipliers.
