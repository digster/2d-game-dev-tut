🏎️ What is a Top-Down Racing Sim?

A top-down racing sim is the simplest possible vehicle game — the camera looks straight down, the car is a small rectangle that points somewhere, and the math of how it moves is the entire show. There's no 3D, no perspective tricks, no shaders to hide behind. That makes it the perfect place to learn how cars actually move.

🕹️ Games that lean on this loop

  • Micro Machines — the granddaddy of top-down racers
  • Death Rally / Mashed — top-down arcade racing with weapons
  • Race Driver: GRID and many indie sims — overhead camera modes
  • The original Need For Speed — early demo modes used top-down for pace cars
  • REDLINE (Sega 1999) — an early sim-leaning top-down racer

Sim vs arcade — what we're choosing

Two design extremes:

  • Arcade: the car turns instantly, accelerates linearly, never slides. Fun in 30 seconds, hides all the physics.
  • Sim: the car has separate heading (where it's pointed) and velocity (where it's going), and they only line up when the tires grip. Slides happen when grip runs out. Acceleration depends on what tires can transmit.

This track picks sim. We start gentle — the Beginner tier has no friction or slip yet — but every choice we make in the integration loop is made with the sim future in mind. You won't have to throw anything away when grip arrives in Intermediate.

💡 The whole game is one loop: read input → update car state → draw car. Once we have those three pieces working, every later tier slots into "update car state" — friction goes there, walls go there, the AI driver goes there.

📐 A Car = Position + Heading + Velocity

Forget cars for a moment. From the simulator's point of view, a car is just three numbers — well, three pieces of state — that evolve over time:

  • Position — a 2D vector (x, y) in pixels. Where the car is.
  • Heading — a single angle in radians. Where the car is pointed.
  • Velocity — a 2D vector in pixels-per-second. Where the car is moving.

🤔 Why separate heading and velocity?

In an arcade game, "facing" and "moving" are the same — the car always goes where it points. In a sim, they diverge. A car drifting through a corner is pointed at the apex but moving sideways; that's the whole drama of grip. Even in this Beginner tier where we don't simulate grip yet, we keep them separate so adding it later is one variable, not a rewrite.

A racing-sim car class starts as little as this:

// The three pieces of state every racing sim needs.
// Heading is in radians; velocity is in pixels-per-second.
class Car {
    constructor(x = 0, y = 0) {
        this.position = new Vector2D(x, y);   // px
        this.velocity = new Vector2D(0, 0);   // px/s
        this.heading  = 0;                    // rad (0 = facing +X = "right")
    }

    // Speed is just |velocity|. Worth a getter so demos can show it cheaply.
    get speed() {
        return this.velocity.length();
    }
}

🎮 Throttle and Steering as Input

Player input maps to two numbers per frame: throttle in the range [-1, +1] (forward / brake-or-reverse) and steering in the range [-1, +1] (left / right). On a keyboard:

  • W or ↑ → throttle = +1
  • S or ↓ → throttle = −1 (brake; reverses once you're stopped)
  • A or ← → steering = −1
  • D or → → steering = +1
  • Nothing pressed → 0

📐 The integration loop, in words

Every frame we do these four things in order, multiplying every rate by dt:

  1. Convert steering into a change in heading (heading += steer · steerRate · dt). But: only when the car is actually moving. Cars can't pivot in place.
  2. Convert throttle into an acceleration along the car's heading, and add that to velocity.
  3. Move position by velocity × dt.
  4. Clamp velocity to a maximum speed so the demo doesn't run off the canvas.

That's the whole loop. Friction, walls, AI — everything later in the track adds one more line in there.

📝 Click to see: the integration loop in code 
// Tunables. We pick numbers that look right at 800x500 canvas size.
const MAX_SPEED = 280;     // px/s
const MAX_ACCEL = 220;     // px/s^2
const MAX_STEER_RATE = 2.4; // rad/s at full lock

// dt is in seconds. Pass throttle and steer as normalized [-1, 1] values.
function update(car, throttle, steer, dt) {
    // 1. Steering changes heading — but only when the car is moving.
    //    Below ~30 px/s, response ramps from 0 to full. Stops "pivot in place" feel.
    const steerScale = clamp(car.speed / 30, 0, 1);
    car.heading += steer * MAX_STEER_RATE * steerScale * dt;

    // 2. Throttle adds acceleration along the car's heading.
    const ax = Math.cos(car.heading) * throttle * MAX_ACCEL;
    const ay = Math.sin(car.heading) * throttle * MAX_ACCEL;
    car.velocity.x += ax * dt;
    car.velocity.y += ay * dt;

    // 3. Move position by velocity. (Position is in px, velocity is in px/s.)
    car.position.x += car.velocity.x * dt;
    car.position.y += car.velocity.y * dt;

    // 4. Clamp speed.
    const sp = car.velocity.length();
    if (sp > MAX_SPEED) car.velocity.multiply(MAX_SPEED / sp);
}
// Tunables. We pick numbers that look right at 800x500 canvas size.
const MAX_SPEED: number = 280;       // px/s
const MAX_ACCEL: number = 220;       // px/s^2
const MAX_STEER_RATE: number = 2.4;  // rad/s at full lock

function update(car: Car, throttle: number, steer: number, dt: number): void {
    const steerScale: number = clamp(car.speed / 30, 0, 1);
    car.heading += steer * MAX_STEER_RATE * steerScale * dt;

    const ax: number = Math.cos(car.heading) * throttle * MAX_ACCEL;
    const ay: number = Math.sin(car.heading) * throttle * MAX_ACCEL;
    car.velocity.x += ax * dt;
    car.velocity.y += ay * dt;

    car.position.x += car.velocity.x * dt;
    car.position.y += car.velocity.y * dt;

    const sp: number = car.velocity.length();
    if (sp > MAX_SPEED) car.velocity.multiply(MAX_SPEED / sp);
}

There's no friction. If you stop pressing throttle, the car coasts forever — Newton's first law, no air, no tires sticking. The next tier adds friction. Right now we want you to feel the bare loop so you appreciate what friction adds.

⏱️ Why We Multiply by dt

The single most important word in every line of the integration loop is dt. It's the time elapsed since the previous frame, measured in seconds. requestAnimationFrame fires roughly 60 times per second on a healthy laptop — so dt is roughly 1/60 ≈ 0.0167s — but it varies. Tab in the background? dt spikes. Hot GPU? Frame drops. A 144Hz monitor? dt halves.

If you forgot the * dt and wrote position.x += velocity.x instead, your car would move per frame rather than per second. Result: it'd literally drive twice as fast on a 144Hz monitor as on a 60Hz one. The demo below proves it. The two cars share the same throttle and steering input; the only difference is one multiplies by dt and one doesn't.

Both cars start at the same position with the same input. Watch them diverge as you change frame rate.
💡 The rule: every line that says "per second" gets a * dt. Position uses velocity (px/s) so position += velocity * dt. Velocity uses acceleration (px/s²) so velocity += accel * dt. Heading uses angular velocity (rad/s) so heading += angVel * dt. Whenever you add a new mechanic — friction, spring forces, anything — read its units and you'll know where dt goes. Deep dive in the Fundamentals: Delta Time & Fixed Timestep.
📝 Click to see: clamping dt on tab refocus 
// When you tab away and come back, requestAnimationFrame can deliver
// a "dt" of 0.5s or more. Without a clamp, the car teleports across the
// canvas (and possibly through walls in later tiers).
const MAX_DT = 0.05; // seconds; ~20fps worst-case

let lastTime = performance.now();
function frame(now) {
    const rawDt = (now - lastTime) / 1000;
    const dt = Math.min(rawDt, MAX_DT);
    lastTime = now;
    update(car, throttle, steer, dt);
    render();
    requestAnimationFrame(frame);
}
requestAnimationFrame(frame);

🎨 Drawing a Rotating Car Sprite

Drawing the car body is two canvas calls: translate to move the origin to the car, then rotate to spin the world around it. Any shape you draw after that — a rectangle, a polygon, an imported sprite — comes out rotated about the car's center, exactly the way you'd expect a car to look when steering.

📐 The two-line pattern

ctx.translate shifts the entire coordinate system; subsequent draws happen relative to that shifted origin. ctx.rotate applies a rotation to the same system. Order matters: translate first, then rotate. Reverse them and you get a car that orbits the canvas origin like a planet.

Under the hood it's a 2D matrix multiplication on the canvas state. Refresher: Matrix Transformations.

📝 Click to see: drawCar 
// Draws a top-down car body of size w x h, centered on the car's position
// and rotated so its long axis points along `heading`.
function drawCar(ctx, car, w = 28, h = 14) {
    ctx.save();
    ctx.translate(car.position.x, car.position.y);
    ctx.rotate(car.heading);

    // Body — a rounded rectangle works as a visible "car" silhouette.
    ctx.fillStyle = '#ef5350';
    ctx.beginPath();
    ctx.roundRect(-w / 2, -h / 2, w, h, 3);
    ctx.fill();

    // Windshield — a darker rectangle near the front to show which way is forward.
    ctx.fillStyle = '#263238';
    ctx.fillRect(w / 6, -h / 2 + 2, w / 4, h - 4);

    ctx.restore();
}
function drawCar(
    ctx: CanvasRenderingContext2D,
    car: Car,
    w: number = 28,
    h: number = 14
): void {
    ctx.save();
    ctx.translate(car.position.x, car.position.y);
    ctx.rotate(car.heading);

    ctx.fillStyle = '#ef5350';
    ctx.beginPath();
    ctx.roundRect(-w / 2, -h / 2, w, h, 3);
    ctx.fill();

    ctx.fillStyle = '#263238';
    ctx.fillRect(w / 6, -h / 2 + 2, w / 4, h - 4);

    ctx.restore();
}

🤔 Try this yourself

Why is the forward direction (cos heading, sin heading)? Because heading is the angle from the +X axis, and standard polar-to-cartesian conversion is exactly that. If you came from a tutorial that uses (sin, -cos) for forward, that tutorial is measuring heading from the +Y axis (compass north) instead — a different convention, same math. 🏎️ Mini-Project: Parking-Lot Sandbox

Time to put it all together. The mini-project combines everything from this tier into one playable sandbox: a top-down car, a parking-lot floor, and persistent skid marks that record where the rear of the car has been. Drive in circles, do figure-eights, scribble your initials.

🎨 The skid-mark trick

The skid marks are drawn to a separate offscreen canvas the same size as the main canvas. Each frame we draw two small dots at the car's rear wheels onto the offscreen layer, then composite the offscreen onto the main canvas before drawing the car. The offscreen never resets unless you press Clear skid marks, so marks persist across frames cheaply — no array of points growing forever.

A subtle fade (globalAlpha 0.995 on a transparent black overlay every few frames) means very old marks gracefully dissolve, which doubles as a nicer look.

📝 Click to see: the offscreen skid-mark layer 
// Build the offscreen canvas once. It's the same size as the visible one.
const skidLayer = document.createElement('canvas');
skidLayer.width  = canvas.width;
skidLayer.height = canvas.height;
const skidCtx = skidLayer.getContext('2d');

// Compute the world positions of the two rear wheels.
// The car's local rear-wheel offsets are (-w/2, +/- h/2). We rotate by heading.
function stampSkidMarks(car, w = 28, h = 14) {
    const cos = Math.cos(car.heading);
    const sin = Math.sin(car.heading);
    for (const side of [-1, +1]) {
        const lx = -w / 2;
        const ly = side * (h / 2);
        const wx = car.position.x + lx * cos - ly * sin;
        const wy = car.position.y + lx * sin + ly * cos;
        skidCtx.fillStyle = 'rgba(20, 20, 20, 0.4)';
        skidCtx.beginPath();
        skidCtx.arc(wx, wy, 1.6, 0, Math.PI * 2);
        skidCtx.fill();
    }
}

// In the render loop:
function render() {
    clearCanvas(ctx, canvas.width, canvas.height); // background asphalt
    drawParkingLot(ctx);
    ctx.drawImage(skidLayer, 0, 0);                // composite the persistent layer
    drawCar(ctx, car);
}
const skidLayer: HTMLCanvasElement = document.createElement('canvas');
skidLayer.width  = canvas.width;
skidLayer.height = canvas.height;
const skidCtx: CanvasRenderingContext2D = skidLayer.getContext('2d')!;

function stampSkidMarks(car: Car, w: number = 28, h: number = 14): void {

    const sin: number = Math.sin(car.heading);
