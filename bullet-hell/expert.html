📈 Why scale is its own problem

Everything so far used the engine's BHField — an array of bullet objects. That's perfect for learning and fine for a few hundred bullets. But a real danmaku puts thousands to tens of thousands of bullets on screen, and at that scale three costs you ignored start to dominate every frame:

  • Allocation & garbage collection — making a new object (or even a temp Vector2D) per bullet per frame floods the GC, which then pauses your game.
  • Per-item work — testing the player against every bullet, or issuing a draw call per bullet, is fine at 200 and fatal at 10,000.
  • Memory layout — scattered objects thrash the CPU cache; packed arrays scream.
💡 The engine stays simple; the lessons go inline. We deliberately do not rewrite the engine's friendly object-based BHField — Beginner→Advanced teach against its object API. Instead these demos build the production store (a pool, a Struct-of-Arrays, a spatial hash) inline, so you can compare them head to head with a live frame-time meter. "Make it work, then make it fast" — in that order.

🐌 The problem — allocations & splice

Here's the naive store: an array of bullet objects, each holding Vector2D position and velocity. Every step allocates a temporary vector per bullet (the vel.copy().multiply(dt) you've used all track), and culling uses splice. Drag the count up and watch the work-ms meter (top-right) climb and spike — those spikes are the garbage collector.

for (let i = naive.length - 1; i >= 0; i--) {
  const b = naive[i];
  b.pos.add(b.vel.copy().multiply(dt));  // temp allocation per bullet
  if (offscreen(b)) naive.splice(i, 1);  // O(n) removal
}
Garbage you never see is still garbage. vel.copy() looks innocent, but at 10k bullets it's 10k allocations per frame — 600k/second. The GC eventually stops the world to clean up, and that's the stutter. The fix is to stop allocating in the hot loop at all.

♻️ The object pool — reuse, don't garbage

An object pool keeps a free-list of dead bullets and hands them back out instead of allocating new ones. Pair it with plain number fields (x/y/vx/vy, no Vector2D) integrated in place, and the hot loop allocates nothing. Same count as before — watch "new allocations last second" sit at zero, and the meter flatten.

class BHBulletPool {
  constructor() { this.live = []; this.free = []; }
  spawn(x, y, vx, vy, r) {
    const b = this.free.pop() || { x:0, y:0, vx:0, vy:0, r:0 };  // reuse, or make ONE
    b.x = x; b.y = y; b.vx = vx; b.vy = vy; b.r = r;
    this.live.push(b); return b;
  }
  step(dt) {
    for (let i = this.live.length - 1; i >= 0; i--) {
      const b = this.live[i];
      b.x += b.vx * dt; b.y += b.vy * dt;          // in place — no temp objects
      if (offscreen(b)) {
        this.live[i] = this.live[this.live.length - 1]; this.live.pop(); // swap-remove
        this.free.push(b);                          // recycle for the next spawn
      }
    }
  }
}
interface PBullet { x:number; y:number; vx:number; vy:number; r:number; }
class BHBulletPool {
  live: PBullet[] = []; free: PBullet[] = [];
  spawn(x:number,y:number,vx:number,vy:number,r:number): PBullet {
    const b = this.free.pop() ?? { x:0,y:0,vx:0,vy:0,r:0 };
    b.x=x; b.y=y; b.vx=vx; b.vy=vy; b.r=r; this.live.push(b); return b;
  }
  step(dt:number): void { /* in-place integrate + swap-remove → free list */ }
}
Pool + plain numbers = a flat steady state. Two changes killed the GC: reusing objects (no new in the loop) and dropping Vector2D for raw fields (no temp vectors). The pool's memory grows to the peak bullet count once, then never again.

🧮 Struct-of-Arrays — flat Float32Arrays

The pool removed allocations, but the data is still scattered objects. The fastest layout is Struct-of-Arrays: instead of an array of {x,y,vx,vy} objects, keep separate flat Float32Arrays — all the x's together, all the y's together. The CPU streams contiguous memory, and culling becomes an O(1) swap-remove. Push this to 10,000 bullets and the step still costs a fraction of a millisecond.

class BHSoaBullets {
  constructor(cap) {
    this.x = new Float32Array(cap); this.y = new Float32Array(cap);
    this.vx = new Float32Array(cap); this.vy = new Float32Array(cap);
    this.r = new Float32Array(cap); this.count = 0;
  }
  step(dt) {
    let n = this.count;
    for (let i = 0; i < n; i++) {
      this.x[i] += this.vx[i] * dt; this.y[i] += this.vy[i] * dt;
      if (offscreen(this.x[i], this.y[i])) {
        n--;                                  // O(1) swap-remove: last live → slot i
        this.x[i] = this.x[n]; this.y[i] = this.y[n];
        this.vx[i] = this.vx[n]; this.vy[i] = this.vy[n]; this.r[i] = this.r[n];
        i--;                                  // re-test the bullet we just moved in
      }
    }
    this.count = n;
  }
}
class BHSoaBullets {
  x: Float32Array; y: Float32Array; vx: Float32Array; vy: Float32Array; r: Float32Array;
  count = 0;
  constructor(cap: number) { this.x = new Float32Array(cap); /* …y, vx, vy, r… */ }
  step(dt: number): void { /* integrate in place; swap-remove on cull */ }
}
Order doesn't matter, so swap-remove is free. Bullets have no meaningful order, so to delete index i we copy the last live bullet into i and shrink the count — no shifting, no allocation. splice would shift everything after i down by one; at 10k that's the difference between smooth and stutter.

🗂️ Spatial hashing — N hit-checks → ~k

Even storage solved, one thing is still O(N): checking the player against every bullet each frame. A uniform spatial hash buckets bullets by grid cell, so the hit test only looks at the 3×3 cells around you — a couple dozen candidates instead of thousands. Click, move, and toggle the hash to watch "checks this frame" collapse.

class BHSpatialHash {
  constructor(cell) { this.cell = cell; this.map = new Map(); }
  clear() { this.map.clear(); }
  insert(i, x, y) {
    const k = Math.floor(x/this.cell) + ',' + Math.floor(y/this.cell);
    (this.map.get(k) || this.map.set(k, []).get(k)).push(i);
  }
  queryNear(x, y, out) {                       // 3x3 cells around (x, y)
    out.length = 0;
    const cx = Math.floor(x/this.cell), cy = Math.floor(y/this.cell);
    for (let gx = cx-1; gx <= cx+1; gx++)
      for (let gy = cy-1; gy <= cy+1; gy++) {
        const a = this.map.get(gx + ',' + gy);
        if (a) for (const idx of a) out.push(idx);
      }
    return out;
  }
}
// per frame: hash.clear(); insert all bullets; only test hash.queryNear(player).
class BHSpatialHash {
  cell: number; map = new Map<string, number[]>();
  constructor(cell: number) { this.cell = cell; }
  insert(i: number, x: number, y: number): void { /* bucket by cell key */ }
  queryNear(x: number, y: number, out: number[]): number[] { /* gather 3x3 */ return out; }
}
Be honest about the build cost. Rebuilding the hash is itself O(N), so for a single query point (one player) it's roughly break-even. The big win comes when you do many queries against it — bullet-vs-bullet, several players, or homing enemies — where you pay O(N) to build once and save on every query after.

🎨 Render batching — one fill, not thousands

The last cost is drawing. Each beginPath/fill has fixed overhead, so doing one per bullet is thousands of state changes a frame. Instead, add every bullet's arc to one path and fill() once (a second pass for the white cores). Toggle batching and watch the draw-ms meter.

// BATCHED: one path for every coloured rim, one for every white core → 2 fills total.
ctx.fillStyle = color; ctx.beginPath();
for (let i = 0; i < S.count; i++) {
  ctx.moveTo(S.x[i] + S.r[i], S.y[i]);     // moveTo before arc → no connecting line
  ctx.arc(S.x[i], S.y[i], S.r[i], 0, Math.PI * 2);
}
ctx.fill();

// UNBATCHED (the slow way): state change + fill PER bullet → 2·N fills.
for (let i = 0; i < S.count; i++) {
  ctx.beginPath(); ctx.arc(S.x[i], S.y[i], S.r[i], 0, Math.PI*2);
  ctx.fillStyle = color; ctx.fill();
}
ctx.fillStyle = color; ctx.beginPath();
for (let i = 0; i < S.count; i++) {
  ctx.moveTo(S.x[i] + S.r[i], S.y[i]);
  ctx.arc(S.x[i], S.y[i], S.r[i], 0, Math.PI * 2);
}
ctx.fill();   // one fill for the whole store
The moveTo before each arc matters. Without it, the path draws a thin line connecting the end of one circle to the start of the next. moveTo lifts the "pen" so each arc is its own sub-path — thousands of separate circles in a single fill.
