SIMULATIONS

🎬 Tooling + spectacle

You can build patterns, fight a boss, and run ten thousand bullets. What turns that into a game is the layer around it: the ability to reproduce a run exactly (for replays, debugging, and fairness), to author attacks as data instead of code, and to make every hit feel like something. Then we compose all of it — patterns, boss, phases, bombs, scoring, performance, juice — into one complete fight.

💡 The one rule that makes replays work: a fixed timestep plus a seeded RNG means the simulation is a pure function of (seed, inputs). Record the seed and the per-tick inputs and you can reconstruct the run bit-for-bit. The discipline: only the simulation may touch the seeded RNG — cosmetics (particles, screen shake) use Math.random so they can vary without breaking the replay.

🎲 Determinism — same seed, same run

Two simulations, both firing an RNG-jittered ring. The left always uses a seeded BHRng; the right uses the same seed too — until you tick "desync," which switches it to Math.random(). Watch the "max position diff": 0 while both are seeded (bit-identical), nonzero the instant one goes non-deterministic.

both seeded — identical…
📝 Click to see: a seeded PRNG (mulberry32) 
class BHRng {                       // deterministic: same seed → same stream
  constructor(seed) { this.s = seed >>> 0; }
  reseed(seed) { this.s = seed >>> 0; }
  next() {                          // mulberry32 — fast, good enough for games
    this.s = (this.s + 0x6D2B79F5) >>> 0;
    let t = this.s;
    t = Math.imul(t ^ (t >>> 15), t | 1);
    t ^= t + Math.imul(t ^ (t >>> 7), t | 61);
    return ((t ^ (t >>> 14)) >>> 0) / 4294967296;   // → [0, 1)
  }
  range(a, b) { return a + this.next() * (b - a); }
}
// Use rng.next() for ANYTHING the sim depends on (spread, timing). Never Math.random.
class BHRng {
  s: number;
  constructor(seed: number) { this.s = seed >>> 0; }
  next(): number { /* …mulberry32… */ return v; }
  range(a: number, b: number): number { return a + this.next() * (b - a); }
}
Why not Math.random()? It's unseedable — you can't ask it to repeat a sequence. A seeded PRNG gives you the same stream every time from the same seed, which is the foundation of replays, netcode lockstep (the Netcode track), and reproducible bug reports.

⏺️ Deterministic replay — record & reconstruct

A replay file is tiny: just the seed and the per-tick inputs. Press ● Record, fly around for a few seconds, then ▶ Replay — the run reconstructs exactly, bullets and all, because the sim is a pure function of seed + inputs. The HUD reports the max difference between the recorded path and the replayed one: 0.000.

// RECORD: reset(seed); each tick, capture the input and advance the sim.
function update(dt) {

Replays are almost free. You don't store positions — those are derived. A whole run compresses to one integer (the seed) plus a few bytes of input per frame. That's how arcade danmaku ships replay files measured in kilobytes.

// the editor emits a plain spec; this turns ANY spec back into a pattern.
function bhPatternFromSpec(spec) {
  const { type, n, speed, arc, k } = spec;
  switch (type) {
    case 'ring':   return (e, f) => ring(f, e, n, speed);
    case 'fan':    return (e, f) => fan(f, e, n, arc * Math.PI/180, e.angle, speed);
    case 'aimed':  return (e, f) => fan(f, e, n, arc * Math.PI/180, Math.PI/2, speed);
    case 'spiral': return (e, f) => { /* n-arm ring; emitter.spin set from spec */ };
    case 'rose':   return (e, f) => { /* speed = |cos(kθ)| ring */ };
  }
}

🧃 Juice — hitstop, shake, particles, audio

"Juice" is the feedback that makes an action feel powerful. Shoot the dummy down and toggle each layer to feel its contribution: hitstop (freeze a few frames on impact), screen shake (trauma that decays), particles (a burst of debris), and an audio cue (here a "♪" flash — you'd trigger a sound sample). None of it changes the rules; all of it changes the feel.

click · Z to shoot…
📝 Click to see: hitstop + trauma shake 
// HITSTOP: on a big hit, freeze the SIM for a few frames (cosmetics still run).
function update(dt) {
  particles.step(dt);                       // cosmetics advance during the freeze
  trauma = Math.max(0, trauma - dt * 1.6);
  if (freeze > 0) { freeze -= dt; return; } // ← the sim is paused here
  // …normal sim…
  if (killedSomething) { freeze = 0.12; trauma = Math.min(1, trauma + 0.8); particles.burst(...); }
}
// SHAKE (render-only): translate the canvas by a decaying random offset.
const s = trauma * trauma;                  // square so it falls off fast
ctx.translate((Math.random()-0.5) * s * 22, (Math.random()-0.5) * s * 22);
let freeze = 0, trauma = 0;
if (freeze > 0) { freeze -= dt; return; }   // hitstop pauses the sim
const s = trauma * trauma;                   // trauma² → shake amount (render-only)
Trauma, not shake. Storing a trauma value (0–1) that decays, and using trauma² for the shake amount, makes shakes add up smoothly and fall off fast — far better than setting a raw shake that lingers. It's the standard "screenshake" recipe.

✦ Bullet-cancel — turn danger into points

Cave's signature mechanic: when you defeat an enemy or end a spell card, every bullet on screen cancels into a score item — the wall of death becomes a shower of points. Let the spiral fill the screen, then press Cancel and watch the danger turn gold. It's pooling in reverse: bullets despawn, items spawn.

let it fill, then cancel…

function cancel() {
  for (const b of enemy.bullets)
    items.push({ x: b.pos.x, y: b.pos.y, vy: -120 - Math.random()*90, age: 0, life: 0.9 });
  score += enemy.count * 50;     // every cancelled bullet is points
  enemy.clear();                 // the screen-clear — danger gone, reward shown
}
interface Item { x: number; y: number; vy: number; age: number; life: number; }

Risk becomes reward. Bullet-cancel is why expert players fly into danger in Cave shooters: a denser screen at the cancel moment is a bigger payout. The same mechanic rewards a no-bomb spell-card capture — which the capstone uses next.

Everything, at once. update(dt) {
