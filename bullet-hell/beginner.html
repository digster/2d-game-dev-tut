🧱 What a danmaku really is

"Danmaku" means barrage — a screen so full of enemy fire it looks impossible, until you realize the gaps are wide and the whole thing is a moving puzzle. Strip away the art and a bullet-hell game is three pieces, sixty times a second:

  • A player you move with surgical precision — because the part of you that can be hit is a single pixel-sized dot, not the whole ship.
  • Bullets — each just a position and a velocity, spawned, moved, and thrown away when it leaves the screen. Our BHField does exactly that and nothing more.
  • One tiny test — did a bullet touch the dot (you die) or merely skim it (you graze, and score)? This is the lesson, so we write it ourselves, on top of the field.

That's the whole skeleton. Spirals, spell-card bosses, ten thousand bullets at once — everything later in this track is more bullets and cleverer ways to spawn them on this same field. So we build the player and the bullet first, then the hit/graze test, each demo adding exactly one idea.

💡 Why a fixed timestep? If bullets moved by velocity × (time since last frame), a pattern would look different on a 30 Hz laptop than a 240 Hz monitor, and a frame hitch could teleport a bullet through your hitbox. Instead we advance the sim in fixed 1/60-second steps and render whenever we can. The engine's bhLoop already does this — and it's why a recorded run can be replayed bit-for-bit later (the Simulations tier's replay feature).

🧰 The engine, loaded on every page in this track

  • BHField + BHBullet — the bullet store and integrator (spawn, move, cull; no collision — that's our job below).
  • bhLoop — the fixed-timestep loop. bhInstallKeys — canvas-scoped player input (move / focus / shoot / bomb).
  • BH + bhDrawField / bhDrawPlayer / bhDrawBullet — the shared palette and asset-free drawing.

Open the track index and scroll to the Scaffold Self-Check to see the field proven — a bullet integrates a known distance, gets culled, and two seeded runs match bit-for-bit.

⚠️ Two traps worth knowing up front: (1) our Vector2D (from the shared utils) mutates in place — v.add(...), v.multiply(...) change v; only v.subtract(...) / v.copy() return new. A bullet owns its own vel, so mutating that is fine, but never scale a shared vector without .copy(). (2) Input is per-canvas: click a demo to focus it, then the arrow keys drive it (and won't scroll the page).

🛰️ The playfield & the fixed loop

Here's the stage: a tall, portrait playfield with a scrolling starfield for a sense of motion. Click the canvas to give it focus, then fly with the arrow keys (or WASD). The loop calls our update(dt) at a fixed 60 Hz and clamps the ship inside the walls — that's all "the game running" means so far.

click the playfield to start…
📝 Click to see: the whole loop is move-then-clamp 
const keys = bhInstallKeys(canvas);   // canvas-scoped: click to focus
const player = { pos: new Vector2D(cx, cy), radius: 3, grazeR: 16 };

function update(dt) {                  // dt is ALWAYS 1/60 — fixed
  // moveDir() is a fresh, normalized vector — safe to mutate.
  player.pos.add(keys.moveDir().multiply(SPEED * dt));
  // keep the ship inside the playfield walls
  player.pos.x = clamp(player.pos.x, bounds.x + PAD, bounds.x + bounds.w - PAD);
  player.pos.y = clamp(player.pos.y, bounds.y + PAD, bounds.y + bounds.h - PAD);
  keys.endFrame();                     // clear one-shot edges (shoot/bomb)
}
bhLoop(update, render).start();        // fixed-step update + per-frame render

Why canvas-scoped input? A bullet-hell tier page stacks many keyboard demos on one scroll. If every demo listened on the window, one press of the arrow key would move all the ships at once and scroll the page. So each canvas is focusable, only swallows the arrow/space keys while focused, and drops all held keys when focus leaves.

🎯 Precise movement — diagonals & focus mode

Danmaku lives or dies on movement precision, and two details matter. First, normalize diagonals: pressing Up+Right gives the raw vector (1, 1), whose length is √2 ≈ 1.41 — so a diagonal would be 41% faster than a straight line. Turn the toggle off and watch the speed read-out jump on diagonals. Second, focus mode: hold Shift to slow down for threading tight gaps — which is also when the game reveals your hitbox.

player.focused = keys.focus;                  // reveals hitbox + graze ring
player.pos.add(dir.multiply(speed * dt));

Focus mode is a genre fundamental. Touhou and Cave both bind a "focus" key that halves your speed and draws the otherwise-hidden hitbox, so you can see exactly the pixels you're threading. We get it almost for free: a speed multiplier plus player.focused, which bhDrawPlayer already uses to reveal the boxes.

•➔ The bullet as a moving point

A bullet is the simplest object in the game: a position and a velocity. The interesting part is how we aim it — a velocity is a polar coordinate, an angle plus a speed, and Vector2D.fromAngle(θ, speed) turns that into (vx, vy). Set the angle and speed, then press Z (after clicking the canvas) or the Fire button. Each bullet flies straight and is culled the instant it leaves the field — watch the live count.

function fire(angleDeg, speed) {

💠 Hitbox vs. graze-box — the genre's whole identity

Here's the trick that defines bullet-hell. Your ship looks big, but only a tiny dot at its centre — the hitbox — can actually be killed. A larger ring around it — the graze-box — rewards you for skimming bullets as close as possible without dying. Both are just circles, and "did they touch?" is a squared-distance comparison.

Fly into the slow blue rain below. Hold Shift to focus and reveal your boxes, then thread bullets right past the dot — each one you skim turns gold and ticks the graze counter. Touch the dot and you'd be dead.

click the playfield, hold Shift, and skim some bullets…
📝 Click to see: hit & graze are two squared-distance tests 
// Squared distance — no Math.sqrt, because we only compare to a squared radius.
// At 10k bullets (Expert tier) skipping the sqrt is the whole performance game.
function bhDist2(a, b) { const dx = a.x - b.x, dy = a.y - b.y; return dx*dx + dy*dy; }

function bhHitTest(player, bullet) {                 // touched the dot -> dead
  const r = player.radius + bullet.radius;
  return bhDist2(player.pos, bullet.pos) <= r * r;
}
function bhGrazeTest(player, bullet) {               // in the ring, not the dot
  const d2 = bhDist2(player.pos, bullet.pos);
  const hit = player.radius + bullet.radius;
  const graze = player.grazeR + bullet.radius;
  return d2 > hit * hit && d2 <= graze * graze;
}

// Each bullet grazes ONCE — a one-shot flag, or it would score 60×/second.
if (!b.grazed && bhGrazeTest(player, b)) { b.grazed = true; grazes++; }
function bhDist2(a: Vector2D, b: Vector2D): number {
  const dx = a.x - b.x, dy = a.y - b.y; return dx*dx + dy*dy;
}
function bhHitTest(player, bullet): boolean {
  const r = player.radius + bullet.radius;
  return bhDist2(player.pos, bullet.pos) <= r * r;
}
function bhGrazeTest(player, bullet): boolean {
  const d2 = bhDist2(player.pos, bullet.pos);
  const hit = player.radius + bullet.radius;
  const graze = player.grazeR + bullet.radius;
  return d2 > hit * hit && d2 <= graze * graze;
}
The hitbox is a design knob, not a constant. A 2–3px hitbox under a 24px sprite is what makes a wall of bullets fair: your eye reads the big ship, but the game judges the tiny dot. Shrink it and the game gets cruel; grow it and the danmaku stops feeling like danmaku. This single number is most of a bullet-hell game's difficulty.

🌧️ A stream of bullets — a cooldown is a fire-rate

A single bullet isn't a barrage. A stream is just: spawn one bullet every interval seconds. That interval is the fire-rate — drag it down and the rain thickens. We sweep the aim with a sine so it paints a readable ribbon instead of a single column. Weave through it; skimmed bullets turn gold.

fire every 120 ms
click the playfield and weave…
📝 Click to see: a timer that fires when it runs out 
let fireTimer = 0, t = 0;

function update(dt) {
  t += dt;
  fireTimer -= dt;
  // `while`, not `if`: at a tiny interval we may owe MORE than one bullet this
  // step — spawn them all, and carry the remainder so timing never drifts.
  while (fireTimer <= 0) {
    fireTimer += interval;
    const sweep = Math.sin(t * 2.2) * 0.5;              // ± rad off straight-down
    const v = Vector2D.fromAngle(Math.PI / 2 + sweep, 210);
    field.spawn(enemy.x, enemy.y, v.x, v.y, { radius: 5, color: BH.bullet });
  }
  field.step(dt);
}
let fireTimer = 0, t = 0;
function update(dt: number): void {
  t += dt; fireTimer -= dt;
  while (fireTimer <= 0) {
    fireTimer += interval;
    const sweep = Math.sin(t * 2.2) * 0.5;
    const v = Vector2D.fromAngle(Math.PI / 2 + sweep, 210);
    field.spawn(enemy.x, enemy.y, v.x, v.y, { radius: 5, color: BH.bullet });
  }
  field.step(dt);
}
Why while and not if? At a 40 ms interval and a 16.6 ms step you sometimes owe more than one bullet per step. The while loop with a carried remainder (fireTimer += interval, not = interval) keeps the fire-rate exact regardless of timestep — the same accumulator idea as the game loop itself. In the next tier this timer graduates into a reusable emitter.

🏆 Mini-Project: "First Dodge"

Everything on one playfield. An enemy fires a 3-shot fan that tightens as the clock runs; survive 20 seconds on three lives. Getting hit costs a life, grants a moment of mercy invulnerability, and clears the screen; every bullet you graze is +100. This is a real (tiny) bullet-hell game — movement, focus, the hitbox/graze test, timed spawning and a win/lose loop, composed.
