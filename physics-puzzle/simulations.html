🧮 Scaling up — and the payoff physics it buys

Soft bodies, ragdolls and fluids all mean many bodies — hundreds of particles, dozens of jointed parts. Two pieces of engineering make that affordable, and then three classic effects fall out:

  • A spatial hash kills the O(n²) "test every pair" cost — only bodies sharing a grid cell can touch.
  • Sleeping stops simulating bodies that have come to rest until something disturbs them.
  • Then: soft bodies (a closed Verlet mesh with internal pressure), ragdolls (just the Advanced tier's jointed rigid bodies), and fluid (a swarm of particles whose pressure floats what's submerged).
💡 Both physics families, side by side. This tier runs the position-based family (Verlet — soft bodies, from engine/constraints.js) and the velocity/impulse family (rigid — ragdolls, from engine/rigid.js) at once. The capstone makes them cooperate: rigid blocks topple into a particle fluid and the displaced water floats a duck to the finish.

🧰 What the engine looks like now (5 modules)

  • world.js (the Beginner integrator), loop.js, render.js — the scaffold.
  • rigid.js — rigid bodies + SAT + joints (promoted from Advanced; reused here for ragdolls & the capstone).
  • constraints.js — the Verlet core (promoted from Intermediate; reused here for soft bodies).

This tier's own new code — the spatial hash, sleeping, the pressure soft-body and the particle fluid — stays inline: it's the terminal tier, so there's no later consumer to promote to.

🔲 Spatial hash — O(n²) → ~O(n)

Testing every pair of bodies is n(n-1)/2 checks — 200 balls is ~20,000 per frame, and it explodes from there. A spatial hash drops each body into a grid cell and only tests it against bodies in the same and neighbouring cells. Toggle the grid and watch the pair-test count collapse as you add balls.

class PZSpatialHash {
  constructor(cell) { this.cell = cell; this.map = new Map(); }
  insert(o) {                         // bucket o by its grid cell
    const k = key(o.pos.x / cell | 0, o.pos.y / cell | 0);
    (this.map.get(k) ?? this.map.set(k, []).get(k)).push(o);
  }
  nearby(o) {                         // everything in the 3×3 cells around o
    const out = [];
    for (let dx = -1; dx <= 1; dx++) for (let dy = -1; dy <= 1; dy++)
      for (const x of this.map.get(key(...)) ?? []) out.push(x);
    return out;
  }
}
// collision becomes: for each ball, only test pzBallPair against nearby(ball)
// Same idea the platformer's SpatialGrid uses. The cell size should be ~the
// largest body's diameter, so a body only ever spans a 3×3 neighbourhood. Pairs
// tested drops from n²/2 to roughly n × (avg bodies per cell) — near-linear.

😴 Sleeping bodies & islands

A settled pile is still being simulated every frame for no visible change. Sleeping fixes that: a body whose speed stays below a threshold for a moment goes to sleep — it stops integrating and dims. A moving body that touches a sleeper wakes it (and, through contacts, its whole island). Click to drop a ball into the pile and watch it ripple awake.

click to drop a ball…
Why it matters: sleeping is what lets a game leave hundreds of settled crates lying around at almost no cost — only the handful that are actually moving get simulated. Waking "islands" (groups connected by contact) together avoids a sleeper hanging in mid-air when its support is knocked out.

🫧 Soft bodies — a closed Verlet mesh + pressure

A soft body is the Intermediate rope tier, bent into a loop. Take a ring of Verlet points joined by perimeter constraints (the membrane), then add one force: pressure. Measure the enclosed area each step; if it's smaller than the rest area, push every edge outward. The blob holds its round shape, squishes when you press it into a wall, and springs back. Drag it; drop the pressure and it deflates like a sad balloon.

📝 Click to see: the pressure force 
// integrate the points (Verlet) and relax the perimeter constraints, then:
const area = polygonArea(points);
const pressure = (restArea / area - 1) * pressureK;   // >0 when compressed
for (const [a, b] of edges) {
  const n = outwardNormal(a, b);          // perpendicular to the edge
  const f = pressure * edgeLength(a, b);  // bigger edges push harder
  a.pos.add(n.scale(f * 0.5));
  b.pos.add(n.scale(f * 0.5));            // inflate the membrane outward
}
// This is the classic 2D "pressurised soft body": a closed mass-spring (here
// Verlet) membrane + an ideal-gas-like pressure term ∝ (restArea/area − 1). The
// same PZVerletPoint / PZConstraint from engine/constraints.js — just wired into
// a loop and inflated.

🧍 Ragdolls — jointed rigid bodies

A ragdoll is the most "advanced-looking" effect in the track and the least new code: it's the Advanced tier's rigid bodies and pivot joints, assembled into a skeleton — a torso, a head, and jointed arms and legs (10 bodies, 9 joints). Every joint just keeps two parts pinned at a shared point, so the whole thing flops under gravity. Drag a hand or a foot to puppet it.

// place the two parts, then pin them at the world point they share (the elbow,
// the hip…). PZJoint wants the anchor in each body's LOCAL frame, so convert:
function pzJointAt(a, b, wx, wy) {
  return new PZJoint(a, worldToLocal(a, wx, wy), b, worldToLocal(b, wx, wy));
}
joints = [
  pzJointAt(torso, head,   cx,      cy - 30),   // neck
  pzJointAt(torso, upperArmL, cx-12, cy - 24),  // shoulder
  pzJointAt(upperArmL, lowerArmL, cx-42, cy-22),// elbow
  /* …hips, knees… */
];
pzStepWorld(bodies, joints, 0, GRAVITY, dt);    // the engine does the rest
function worldToLocal(b: PZRigidBody, wx: number, wy: number) {
  const dx = wx - b.pos.x, dy = wy - b.pos.y;
  const c = Math.cos(-b.angle), s = Math.sin(-b.angle);
  return { x: dx * c - dy * s, y: dx * s + dy * c };
}
// Real games add angle LIMITS to each joint so elbows don't bend backwards; the
// constraint is the same, just clamped.

💧 Fluid & buoyancy — water is many particles

Water is just a swarm of particles that don't like being crowded. Each one pushes neighbours that get too close (that's pressure) and nudges toward their average velocity (that's viscosity) — over the spatial hash, near-linear. Drop something light in and it floats: the dense fluid beneath nets an upward shove. Drag the duck under the surface and let go.

drag the duck under…
Simplified, but honest: this is a position-based fluid — pressure is implicit in neighbour repulsion. The rigorous version uses smoothing kernels (poly6 for density, a spiky gradient for pressure, a viscosity laplacian) — see the SPH demo in simulation-v2. Both say the same thing: water is many tiny bodies, and floating is their pressure pushing up on what's submerged.
