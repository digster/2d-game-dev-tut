🧬 Two families of physics

The Beginner tier stored a velocity on every body and changed it with forces and impulses. That's the impulse / velocity-based family, and it's perfect for things that fly and bounce. But it's clumsy for a rope: a rope is a hard rule — "these beads stay a fixed distance apart" — and chasing that rule with forces makes springs that wobble.

So we switch families. Verlet integration stores each point's previous position instead of a velocity; the velocity is the implicit gap between where a point is and where it just was. The payoff is enormous: a constraint becomes a single line — "if these two points drifted apart, nudge them back." No springs, no tuning.

💡 Both families live in one world. Ropes and soft bodies are Verlet; crates and cannonballs are impulse-based. They cooperate in the grand capstone (the candy you cut here will land on rigid bodies later). For now, everything on this page is Verlet.

🧰 What we build this tier (all inline in intermediate-demos.js)

  • PZVerletPoint — a point with pos + prev; velocity is implicit.
  • PZConstraint — a distance rule between two points; solve() nudges them to rest length.
  • pzStepRope — integrate every point, then relax every constraint K times.
  • pzCutBlade / pzClickCut — sever a constraint with a swipe or a click (uses lineIntersection from the shared utils).

↩️ Verlet vs Euler — store the previous position

Two particles fall under the same gravity. The red one is Euler — it keeps a velocity (vx, vy). The cyan one is Verlet — it keeps its previous position (the faint ghost), and its velocity is just pos − prev. They move identically. Hit Gust: Euler adds to its velocity variable, Verlet shifts its previous position — same result, but Verlet never names a velocity.

both falling…
📝 Click to see: the same step, two ways 
// EULER — store velocity, change it with acceleration:
vy += g * dt;        pos.x += vx * dt;  pos.y += vy * dt;

// VERLET — store the previous position; velocity is implicit:
const vx = pos.x - prev.x;            // = velocity · dt, "for free"
const vy = pos.y - prev.y;
prev.x = pos.x;  prev.y = pos.y;      // remember where we were
pos.x += vx;                          // carry the motion forward...
pos.y += vy + g * dt * dt;            // ...and add gravity as a·dt²
// integrate one Verlet point (no velocity field anywhere):
integrate(gx: number, gy: number, dt: number, damping: number): void {
  if (this.pinned) return;
  const vx = (this.pos.x - this.prev.x) * damping;  // implicit velocity·dt
  const vy = (this.pos.y - this.prev.y) * damping;
  this.prev.x = this.pos.x;  this.prev.y = this.pos.y;
  this.pos.x += vx + gx * dt * dt;
  this.pos.y += vy + gy * dt * dt;
}
Why this matters for ropes: because velocity is implicit, you can just move a point and its velocity updates itself. Constraints do exactly that — move points to satisfy a distance — so they cost nothing extra. That's the whole reason ropes use Verlet.

📏 The distance constraint — nudge to rest length

One peg, one bob, one rule between them: stay 130 px apart. Drag the peg and the bob trails it at a fixed distance; drag the bob aside and let it swing. Solving the constraint is a single nudge — measure how far off the rest length they are, and move each point half of that error toward the other (a pinned point doesn't move, so its partner takes the whole correction).

drag the peg or the bob…
📝 Click to see: solving one constraint 
solve() {
  const dx = b.pos.x - a.pos.x, dy = b.pos.y - a.pos.y;
  const d  = Math.hypot(dx, dy) || 1e-6;
  const diff = (d - this.rest) / d;          // >0 stretched, <0 squashed
  // a pinned endpoint stays put, so its partner takes the full move:
  const aMove = a.pinned ? 0 : (b.pinned ? 1 : 0.5);
  const bMove = b.pinned ? 0 : (a.pinned ? 1 : 0.5);
  a.pos.x += dx * diff * aMove;  a.pos.y += dy * diff * aMove;
  b.pos.x -= dx * diff * bMove;  b.pos.y -= dy * diff * bMove;
}
solve(): void {
  const dx = this.b.pos.x - this.a.pos.x;
  const dy = this.b.pos.y - this.a.pos.y;
  const d  = Math.hypot(dx, dy) || 1e-6;
  const diff = ((d - this.rest) / d) * this.stiffness;
  const aMove = this.a.pinned ? 0 : (this.b.pinned ? 1 : 0.5);
  const bMove = this.b.pinned ? 0 : (this.a.pinned ? 1 : 0.5);
  this.a.pos.x += dx * diff * aMove; this.a.pos.y += dy * diff * aMove;
  this.b.pos.x -= dx * diff * bMove; this.b.pos.y -= dy * diff * bMove;
}

🪢 Building a rope — iterations are stiffness

A rope is just a row of those constraints, end to end, here pinned at both ends to hang as a cable. One catch: solving each constraint once tugs its neighbours out of place, so a single pass leaves the rope stretchy. The fix is relaxation — solve the whole chain several times per step. Slide the iterations from 1 to 20 and watch the rope go from rubber-band to steel. Grab any bead to yank it.

function pzStepRope(points, constraints, gx, gy, dt, iterations, damping) {
  for (const p of points) p.integrate(gx, gy, dt, damping);   // move everything
  for (let k = 0; k < iterations; k++)                        // then relax the
    for (const c of constraints) c.solve();                   // constraints K times
}
// build: a line of points, a constraint between each neighbour, ends pinned
points[0].pinned = points[N - 1].pinned = true;
for (let i = 0; i < N - 1; i++)
  constraints.push(new PZConstraint(points[i], points[i + 1]));
function pzStepRope(points: PZVerletPoint[], constraints: PZConstraint[],
                   gx: number, gy: number, dt: number,
                   iterations: number, damping: number): void {
  for (const p of points) p.integrate(gx, gy, dt, damping);
  for (let k = 0; k < iterations; k++)
    for (const c of constraints) c.solve();
}
The classic trade-off: more iterations = stiffer and more stable, but more CPU. Games pick the smallest count that looks right (often 3–8). It's the same relaxation idea the rigid-body solver uses in the Advanced tier, and the soft bodies in Simulations.

🎯 Swinging & momentum — timing is everything

A rope pinned at the top with a heavy bob is a pendulum. Drag the bob aside and let go — Verlet keeps the swing energy almost perfectly, so the bob carries momentum. Try to fling it through the gold ring: you can't aim it directly, you have to release at the right point in the arc and let the swing do the rest. That timing is the puzzle in Cut the Rope.

drag the bob & release…
Where does the velocity come from? While you drag, we move the bob and update its prev each frame — so on release, pos − prev already holds your drag velocity. The throw is free, straight out of the Verlet representation.

✂️ The cut — sever a constraint

The signature verb. Three candies hang from ropes; swipe across a rope (or click it) and that constraint is gone — the candy falls free. Cutting is almost insultingly simple: a constraint has a broken flag, and a swipe that crosses the rope segment flips it. Drop all three into the bowl.

swipe a rope to cut it…
📝 Click to see: a cut is one flag + a segment test 
// blade = the segment the pointer swept this frame (lastPos → pos).
// If it crosses a rope segment, break that constraint. (lineIntersection
// is from shared/utils.js — it already returns the crossing point or null.)
function pzCutBlade(constraints, x0, y0, x1, y1) {
  for (const c of constraints) {
    if (c.broken) continue;
    if (lineIntersection({x:x0,y:y0}, {x:x1,y:y1}, c.a.pos, c.b.pos)) {
      c.broken = true;                 // a severed constraint is just skipped
    }
  }
}
function pzCutBlade(constraints: PZConstraint[],
                   x0: number, y0: number, x1: number, y1: number): void {
  for (const c of constraints) {
    if (c.broken) continue;
    if (lineIntersection({x:x0,y:y0}, {x:x1,y:y1}, c.a.pos, c.b.pos))
      c.broken = true;
  }
}
// the solver and renderer both `if (c.broken) continue;` — that's the whole cut.

🎮 Mini-Project: "Deliver"

Everything together. A candy hangs from a rope, a shelf blocks the straight drop, and the goal is off to the right. Drag the candy to build up a swing, then cut the rope at the right moment so it sails over the shelf and into the goal. The candy is a Verlet point that collides with the shelf and walls by simple depenetration — push it out, and its velocity corrects itself.

drag to swing, then cut…
📝 Click to see: Verlet payload-vs-wall is just depenetration 
// No velocity to reflect — Verlet derives it from (pos − prev), so simply
// pushing pos out of the wall kills the inward motion automatically.
function pzVerletBlock(p, block) {
  const cx = clamp(p.pos.x, block.x, block.x + block.w);
  const cy = clamp(p.pos.y, block.y, block.y + block.h);
  const dx = p.pos.x - cx, dy = p.pos.y - cy;
  const d2 = dx * dx + dy * dy;
  if (d2 > p.radius * p.radius) return;        // not touching
  const d = Math.sqrt(d2) || 1e-6;
  p.pos.x += (dx / d) * (p.radius - d);        // push out along the normal
  p.pos.y += (dy / d) * (p.radius - d);
}
function pzVerletBlock(p: PZVerletPoint, block: Rect): void {
  const cx = clamp(p.pos.x, block.x, block.x + block.w);
  const cy = clamp(p.pos.y, block.y, block.y + block.h);
  const dx = p.pos.x - cx, dy = p.pos.y - cy;
  const d2 = dx * dx + dy * dy;
  if (d2 > p.radius * p.radius) return;
  const d = Math.sqrt(d2) || 1e-6;
  p.pos.x += (dx / d) * (p.radius - d);
  p.pos.y += (dy / d) * (p.radius - d);
}

✅ Recap — what you built

  • Verlet integration — store the previous position; velocity is the implicit gap pos − prev.
  • The distance constraint — one nudge toward rest length; a pinned point takes none of it.
  • Ropes — a row of constraints, relaxed K times per step; iterations are stiffness.
  • Swinging — Verlet conserves the arc's energy, so momentum is all about release timing.
  • The cut — a broken flag flipped by a swipe-vs-segment test.
  • "Deliver" — swing + cut + Verlet depenetration to land a payload in the goal.
