🗝️ What makes a roguelike a roguelike

The genre is named after Rogue (1980) and its descendants — NetHack, Brogue, Dungeon Crawl Stone Soup. They look nothing alike on the surface, but they share a skeleton, and two bones of that skeleton are non-negotiable:

  • The world is a grid. Not pixels, not free-floating positions — a rectangular array of tiles, each one wall or floor or stairs. Everything (the player, monsters, items) lives on a tile, at integer coordinates.
  • Time is turn-based. Nothing moves until you do. Press a key, you take one action, then every monster takes one action, then the world waits again. There is no clock ticking in the background — and that single fact rewrites the architecture of everything you've built in the other tracks.
💡 Why this is its own track: every other track in this repo runs a real-time loop — requestAnimationFrame fires 60 times a second whether you touch the keyboard or not, and physics integrates against dt. A roguelike inverts that: the simulation only advances when the player commits an action. We'll build that inversion in the very first two demos, then spend the rest of the track stacking the genre's classic systems on top of it.

🧱 Two helpers, loaded on every page in this track

  • RogueRng — a deterministic random-number generator. Same seed + same actions = same dungeon, same monsters, same dice. (Plus roguelike helpers: dice(2,6), pick(arr), weighted(table).)
  • Level + drawGlyphGrid — the grid data structure (a flat Uint8Array of tile types with isWalkable / isOpaque queries) and the ASCII renderer that paints it onto a canvas as monospace glyphs.

Both shipped in the scaffold commit. Open the track index and scroll to the Scaffold Self-Check at the bottom — it proves both load and renders a little room with the exact glyphs you'll see below.

How the grid is stored

📝 Click to see: a Level is just a flat array of tile types 
// Tile types are small integers so the whole map fits in one Uint8Array.
// WALL = 0 matters: a fresh array is all zeros, i.e. "solid rock" — and
// generators CARVE floor into it (you'll do that in the Intermediate tier).
const Tile = { WALL: 0, FLOOR: 1, STAIRS_DOWN: 2 };

// Build a 19x11 room: solid rock, then hollow out the interior to floor.
const level = new Level(19, 11, Tile.WALL);
for (let y = 1; y < 10; y++)
    for (let x = 1; x < 18; x++)
        level.set(x, y, Tile.FLOOR);
level.set(17, 9, Tile.STAIRS_DOWN);

// The (x, y) -> index math is the single most-used line in a roguelike.
// Reading out of bounds returns WALL, so movement code never special-cases
// the edges — the world is "walled in" by definition.
const i = level.idx(x, y);          // y * width + x
const standable = level.isWalkable(x, y);  // floor / stairs / door, not wall

// The player is NOT in the array — it's a separate object holding a tile coord.
const player = { x: 1, y: 1, ch: '@', color: '#7CF2C8' };
const enum Tile { WALL = 0, FLOOR = 1, STAIRS_DOWN = 2 }

interface Actor { x: number; y: number; ch: string; color: string; }

const level = new Level(19, 11, Tile.WALL);
for (let y = 1; y < 10; y++)
    for (let x = 1; x < 18; x++)
        level.set(x, y, Tile.FLOOR);
level.set(17, 9, Tile.STAIRS_DOWN);

const player: Actor = { x: 1, y: 1, ch: '@', color: '#7CF2C8' };
const standable: boolean = level.isWalkable(player.x, player.y);
💡 The player lives outside the grid. The tile array only stores the terrain. Actors (player, monsters) and items are separate objects that hold a tile coordinate. That separation is what lets two things share knowledge of a tile — "is the floor at (5,5) occupied?" is answered by scanning actors, not by the map.

⏳ The turn loop — turn-based vs real-time

This is the heart of the genre. Below are two copies of the same room, each with a goblin g trying to reach you. Click to focus, then move with the arrow keys / WASD / hjkl and watch the difference:

  • LEFT — real-time: the goblin steps toward you on a timer (about three steps a second) whether you move or not. Sit still and it still closes in.
  • RIGHT — turn-based: the goblin steps exactly once for each step you take. Sit still and it freezes. The world holds its breath until you act.
click, then move — and try sitting still…

🔍 What to notice

Take ten seconds to think between moves. In the real-time room the goblin used those ten seconds to walk right up to you; in the turn-based room it didn't move at all. That pause is the genre's superpower — it's why roguelikes can be brutally tactical: you always have all the time in the world to plan the single best move, and the consequences only unfold when you commit.

📝 Click to see: a real-time loop vs a turn-based loop, side by side 
// ── REAL-TIME (every other track in this repo) ───────────────────
// The simulation advances on a clock, regardless of input.
function frame(now) {
    const dt = (now - last) / 1000; last = now;
    world.update(dt);            // monsters move, physics integrates...
    render();
    requestAnimationFrame(frame);
}
requestAnimationFrame(frame);    // runs forever

// ── TURN-BASED (a roguelike) ─────────────────────────────────────
// The simulation advances ONLY when the player commits an action.
canvas.addEventListener('keydown', (e) => {
    const step = keyToStep(e);   // arrow / WASD / hjkl -> {dx, dy} or null
    if (!step) return;

    const result = tryMove(player, step);   // 'moved' | 'attacked' | 'blocked'
    if (result === 'blocked') return;        // walked into a wall: FREE retry

    // The player spent a turn — now everyone else gets one.
    for (const monster of monsters) takeMonsterTurn(monster);
    turnCount++;
    render();                    // we only redraw when something happened
});
type Step = { dx: number; dy: number };
type MoveResult = 'moved' | 'attacked' | 'blocked';

// Turn-based: input drives the simulation, not a clock.
canvas.addEventListener('keydown', (e: KeyboardEvent) => {
    const step: Step | null = keyToStep(e);
    if (!step) return;

    const result: MoveResult = tryMove(player, step);
    if (result === 'blocked') return;   // no turn spent

    for (const monster of monsters) takeMonsterTurn(monster);
    turnCount++;
    render();
});
💡 The whole genre fits in that second block. "Player acts → if the action was real, everyone else acts → redraw." Everything we add later — pathfinding, items, status effects, speed — slots into takeMonsterTurn or tryMove. Notice we only render() after something changed; there's no background loop burning CPU while you think.

👊 Bump-to-attack — move or hit from one keypress

Roguelikes almost never have a separate "attack" button. Instead, walking into something attacks it. One keypress, and the game decides: is the target tile empty floor? Then move there. Is something standing on it? Then attack that thing instead, and stay put. Click to focus and bump into the training dummies d — each hit rolls 1d6 damage.

click, then bump a dummy…
📝 Click to see: the move-or-attack resolution (the one branch that defines the feel) 
// One keypress -> one of three outcomes. Only the first two cost a turn.
function tryMove(actor, step) {
    const tx = actor.x + step.dx;
    const ty = actor.y + step.dy;

    // 1) Is a creature standing on the target tile? Bump = attack.
    const target = actorAt(tx, ty);
    if (target && target !== actor) {
        attack(actor, target);
        return 'attacked';
    }

    // 2) Is the target tile open floor? Step onto it.
    if (level.isWalkable(tx, ty) && !actorAt(tx, ty)) {
        actor.x = tx;
        actor.y = ty;
        return 'moved';
    }

    // 3) A wall (or the map edge). Nothing happens — and no turn is spent.
    return 'blocked';
}

function attack(attacker, target) {
    const dmg = rng.dice(1, 6);          // 1d6 — deterministic given the seed
    target.hp -= dmg;
    log(`${attacker.name} hits ${target.name} for ${dmg}.`);
    if (target.hp <= 0) { target.dead = true; log(`${target.name} dies!`); }
}
type Step = { dx: number; dy: number };
type MoveResult = 'moved' | 'attacked' | 'blocked';

interface Actor { x: number; y: number; hp: number; name: string; dead?: boolean; }

function tryMove(actor: Actor, step: Step): MoveResult {
    const tx = actor.x + step.dx, ty = actor.y + step.dy;

    const target = actorAt(tx, ty);
    if (target && target !== actor) { attack(actor, target); return 'attacked'; }

    if (level.isWalkable(tx, ty) && !actorAt(tx, ty)) {
        actor.x = tx; actor.y = ty; return 'moved';
    }
    return 'blocked';
}
💡 Walls don't cost a turn; attacks do. That 'blocked' return is doing real work — bumping a wall is a free retry, so monsters don't get a free turn just because you misjudged a corner. Attacking, though, is your action for the turn: you can't hit and step in the same move.

📜 The message log + seeded combat

Bumping a passive dummy is one thing — but a real fight has turn order: you act, then the enemy acts back. And because so much happens off-screen-ish (dice rolls, deaths, misses), every roguelike has a message log narrating it. Here's a tiny duel: you @ (20 HP, 1d6) versus a rat r (8 HP, 1d4). Corner it. 🎲 Why a seeded log matters

Combat is dice, and dice are random — but in this track "random" means RogueRng, not Math.random(). Set the same seed, fight the rat with the same moves, and you'll get the same hits, the same damage, the same death turn, every single time. That reproducibility is what makes a bug filable ("seed 7, move right three times, the rat should be dead") and what powers the shareable "daily challenge" you'll build in the Simulations tier.

📝 Click to see: how the sleeping rat wakes and hunts (no pathfinding yet) 
// The rat's whole brain for this tier. (The Advanced tier replaces the
// "step toward" line with real A* / Dijkstra pathing around walls.)
function takeRatTurn(rat, player) {
    // Asleep until the player is close — a stand-in for the real
    // field-of-view check you'll build in the Advanced tier.
    const dist = Math.abs(rat.x - player.x) + Math.abs(rat.y - player.y); // Manhattan
    if (rat.asleep) {
        if (dist <= rat.senseRange) { rat.asleep = false; log('The rat notices you!'); }
        else return;             // keep snoozing, no action
    }

    // Adjacent? Bite. Otherwise greedily step one tile closer.
    if (dist === 1) { attack(rat, player); return; }
    const step = stepToward(rat, player);     // greedy cardinal step
    tryMove(rat, step);
}

// Greedy one-tile step toward a target, with a simple wall fallback:
// try the bigger axis first, then the other, then give up (wait).
function stepToward(from, to) {
    const dx = Math.sign(to.x - from.x), dy = Math.sign(to.y - from.y);
    const tries = Math.abs(to.x - from.x) > Math.abs(to.y - from.y)
        ? [{dx, dy: 0}, {dx: 0, dy}]
        : [{dx: 0, dy}, {dx, dy: 0}];
    for (const t of tries)
        if (level.isWalkable(from.x + t.dx, from.y + t.dy)) return t;
    return { dx: 0, dy: 0 };     // boxed in — wait this turn
}
interface Rat { x: number; y: number; asleep: boolean; senseRange: number; }

function takeRatTurn(rat: Rat, player: Actor): void {
    const dist = Math.abs(rat.x - player.x) + Math.abs(rat.y - player.y);
    if (rat.asleep) {
        if (dist <= rat.senseRange) { rat.asleep = false; log('The rat notices you!'); }
        else return;
    }
    if (dist === 1) { attack(rat, player); return; }
    tryMove(rat, stepToward(rat, player));
}
💡 "Greedy step toward" is not pathfinding. It walks straight at you and gets stuck on the far side of a wall — try luring the rat around a pillar. Fixing that is exactly what the Advanced tier's A* and Dijkstra maps are for. Starting with the naive version makes the upgrade feel like an upgrade.
