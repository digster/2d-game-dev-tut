🎒 From a fist to an arsenal

Everything so far has been about space — the grid, the dark, the chase. This tier is about stuff: the systems that turn a chase into an RPG.

  • Items as data. A sword and a potion are the same kind of thing — an entity with components — which is what lets one inventory hold anything.
  • Equipment & derived stats. Wearing gear changes numbers you compute, never numbers you store.
  • Status effects & identification. Poison ticks; haste speeds you up; and that unlabelled potion might be either — the risk is the gameplay.
  • Energy & speed. The turn loop stops being "everyone takes one turn" — fast things act more often.
  • Monster minds. A brute, a coward, and an archer want different things; FSMs and behavior trees give them each a brain.
💡 The theme is composition. None of these is a special case bolted on — each is a small system, and the capstone runs all of them at once on top of the Advanced tier's fog and pursuit. That's how real roguelikes get their famous depth.

🗃️ Items as data — a tiny ECS

A weapon, a potion, a scroll: wildly different in play, identical in structure. Each is an entity with components — a glyph, maybe an equip component, maybe a consume component. The inventory doesn't care which; it just holds entities. Click to focus, walk onto items to pick them up (or press g), and d to drop the last one. The panel queries the world by component to label each item.

// An item is an entity: an id, a glyph, and a bag of components.
class Item {
    constructor(name, glyph, color) { this.name = name; this.glyph = glyph; this.color = color; this.comps = {}; }
    add(name, data) { this.comps[name] = data; return this; }   // chainable
    has(name) { return name in this.comps; }
    get(name) { return this.comps[name]; }
}

// Factories compose components — a sword is "equippable + a weapon";
// a potion is "consumable". The inventory holds all of them the same way.
const sword  = new Item('short sword', '/', '#cfd8dc').add('equip', { slot: 'weapon' }).add('weapon', { dice: [1, 8] });
const armor  = new Item('leather armor', '[', '#c98a4b').add('equip', { slot: 'armor' }).add('armor', { def: 1 });
const potion = new Item('murky potion', '!', '#ab47bc').add('consume', { effect: 'heal', power: 8 });

// The "World query" — ask for every item with a given component.
class World {
    constructor() { this.items = []; }
    add(it) { this.items.push(it); return it; }
    query(comp) { return this.items.filter(it => it.has(comp)); }
}
const wearables = world.query('equip');     // [sword, armor]
const drinkables = world.query('consume');  // [potion]
interface Components { equip?: { slot: 'weapon' | 'armor' }; weapon?: { dice: [number, number] }; armor?: { def: number }; consume?: { effect: string; power: number }; }

class Item {
    comps: Components = {};
    constructor(public name: string, public glyph: string, public color: string) {}
    add<K extends keyof Components>(name: K, data: Components[K]): this { this.comps[name] = data; return this; }
    has(name: keyof Components): boolean { return name in this.comps; }
    get<K extends keyof Components>(name: K): Components[K] { return this.comps[name]; }
}
💡 Why ECS for items? A "ring of fire resistance that's also a thrown weapon" isn't a new class — it's an entity with three components. New behaviour = new component, not a deeper inheritance tree. The systems (equip, consume, throw) each just query for the component they care about.

🛡️ Inventory & equipment (derived stats)

You start with a few items. Click to focus, then press an item's number (1, 2, 3…) to equip or unequip it — weapons fill the weapon slot, armor the armor slot. Your attack and defense aren't stored; they're derived from whatever's equipped, recomputed on the fly. Equip the sword and watch your attack die grow.

click, then press 1–9 to equip…

Backpack — press a number to equip/unequip

🧮 Stored vs derived

Store the equipment (which item is in each slot), never the totals. Then attack() = base + weapon's contribution and defense() = base + armor's contribution, computed whenever you need them. Bugs where your stats drift out of sync with your gear simply can't happen — there's only one source of truth.

🧪 Status effects & the identification game

Three unlabelled potions sit in your pack — a murky, a fizzy, and a glowing one. Each colour maps to an effect for this run — heal, poison, or regen — but which is which? Click to focus, press a potion's number to quaff it, and find out (and identify it for good). Then move around: status effects tick every turn — poison bleeds you, regen heals you.

// Status effects are just {kind, turns, power} records on the actor,
// ticked once per turn. New effect = new case here, nothing else.
function tickStatuses(actor, log) {
    for (const s of actor.statuses) {
        if (s.kind === 'poison') { actor.hp -= s.power; log(`Poison bites you for ${s.power}.`); }
        if (s.kind === 'regen')  { actor.hp = Math.min(actor.maxHp, actor.hp + s.power); }
        s.turns--;
    }
    actor.statuses = actor.statuses.filter(s => s.turns > 0);   // drop the expired
}

// Identification: each appearance (colour) maps to a hidden effect, shuffled
// by the seed. Quaffing reveals the mapping for the rest of the run.
function buildPotionTable(rng) {
    const colours = ['murky', 'fizzy', 'glowing'];
    const effects = rng.shuffle(['heal', 'poison', 'haste']);   // seeded permutation
    return colours.map((c, i) => ({ colour: c, effect: effects[i], known: false }));
}
interface Status { kind: 'poison' | 'regen' | 'haste'; turns: number; power: number; }

function tickStatuses(actor: { hp: number; maxHp: number; statuses: Status[] }, log: (s: string) => void): void {
    for (const s of actor.statuses) {
        if (s.kind === 'poison') { actor.hp -= s.power; log(`Poison bites you for ${s.power}.`); }
        if (s.kind === 'regen')  { actor.hp = Math.min(actor.maxHp, actor.hp + s.power); }
        s.turns--;
    }
    actor.statuses = actor.statuses.filter(s => s.turns > 0);
}
💡 Identification is information, not mechanics. The potion's effect never changes — only your knowledge of it does. That's why a seeded shuffle is enough: the danger is the player's uncertainty, and that's a feature you get almost for free.

⚡ Energy & speed — bending turn order

Until now everyone took exactly one turn each round. Real roguelikes use an energy system: every actor banks energy equal to its speed each tick, and acts when it has 100. A snake (speed 200) acts twice for your every move; a zombie (speed 50) acts half as often. Click to focus and take steps — watch the energy bars fill and the action counts diverge.

click, then step — the snake acts twice per your move
📝 Click to see: the energy/speed scheduler 
const ACTION = 100;   // every action costs 100 energy

// After the player acts, advance the world until the player can act again.
// Faster actors cross the 100 threshold more often, so they act more often.
function runScheduler(player, monsters, actOne) {
    player.energy -= ACTION;
    while (player.energy < ACTION) {
        for (const a of [player, ...monsters]) if (!a.dead) a.energy += a.speed;
        for (const m of monsters) {
            while (!m.dead && m.energy >= ACTION) { actOne(m); m.energy -= ACTION; }
        }
    }
    // player.energy is now >= 100 again → wait for input.
}
const ACTION = 100;

interface Actor { energy: number; speed: number; dead?: boolean; }

function runScheduler(player: Actor, monsters: Actor[], actOne: (m: Actor) => void): void {
    player.energy -= ACTION;
    while (player.energy < ACTION) {
        for (const a of [player, ...monsters]) if (!a.dead) a.energy += a.speed;
        for (const m of monsters) while (!m.dead && m.energy >= ACTION) { actOne(m); m.energy -= ACTION; }
    }
}
💡 Speed is just energy-per-tick. Haste? Temporarily raise the player's speed. Slow? Lower it. A "double move" potion, a hasted boss, a sluggish zombie — all one integer. No special-casing the turn loop; the threshold does the work.

🧠 Monster minds — FSM & behavior trees

Three monsters, three different brains. The brute (B) runs a state machine: sleep → chase → attack. The coward (c) is also an FSM but flips to flee when you get close. The archer (a) runs a behavior tree: keep line of sight, hold a 3–5 tile range, and loose an arrow when it can. Click to focus and move — each monster's current state is labelled above it.

📝 Click to see: an FSM monster and a behavior-tree monster 
// FSM: the brute. One function, transitions chosen by distance + sight.
function bruteTurn(m, player, level) {
    const see = losLine(level, m.x, m.y, player.x, player.y).clear;
    const d = manhattan(m, player);
    switch (m.state) {
        case 'sleep': if (see && d <= 8) m.state = 'chase'; break;
        case 'chase':
            if (d === 1) m.state = 'attack';
            else stepToward(m, player);          // walk in
            break;
        case 'attack':
            if (d === 1) attack(m, player);
            else m.state = 'chase';
            break;
    }
}

// Behavior tree: the archer. Selector tries each branch until one succeeds.
const archerTree = new Selector([
    new Sequence([ new Cond(canShoot),  new Act(shootPlayer) ]),   // in range + LOS → shoot
    new Sequence([ new Cond(tooClose),  new Act(stepAway)    ]),   // too close → back off
    new Sequence([ new Cond(hasLos),    new Act(holdRange)   ]),   // keep distance
    new Act(wander),                                              // fallback
]);
archerTree.run(archer);   // each turn
// Behavior-tree nodes return 'ok' | 'fail'. Selector = OR, Sequence = AND.
type Status = 'ok' | 'fail';
class Cond { constructor(private fn: (a: any) => boolean) {}
    run(a: any): Status { return this.fn(a) ? 'ok' : 'fail'; } }
class Act { constructor(private fn: (a: any) => void) {}
    run(a: any): Status { this.fn(a); return 'ok'; } }
class Sequence { constructor(private kids: any[]) {}
    run(a: any): Status { for (const k of this.kids) if (k.run(a) === 'fail') return 'fail'; return 'ok'; } }
class Selector { constructor(private kids: any[]) {}
    run(a: any): Status { for (const k of this.kids) if (k.run(a) === 'ok') return 'ok'; return 'fail'; } }
💡 FSM vs behavior tree: an FSM is perfect when "what am I doing now" fully decides the next step (brute, coward). A behavior tree shines when you want prioritised behaviour — "shoot if you can, else reposition, else wander" — without a tangle of transitions. Most games use both, picking per monster.
