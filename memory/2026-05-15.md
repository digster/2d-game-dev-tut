# 2026-05-15 — Fix: broken TS Copy-Code exports (expert + advanced tiers)

## Context

A suggested task flagged a latent bug in the isometric-strategy Copy-Code
feature's TypeScript export variants. Triaged it, confirmed it was real (and
slightly worse than described), and fixed it.

## Root cause (two distinct bugs)

The TS demo bodies in `bundles-expert.js` / `bundles-advanced.js` are rebuilt
from their JS twins via:

```js
DEMO_CODE_TS.iso_xDemo = `(function xDemo(): void {
    ...typed header lines...
${DEMO_CODE.iso_xDemo.split('\n').slice(K, -2).join('\n')}
})();`;
```

1. **Systematic off-by-one (all 16 demos).** `.slice(K, -2)` drops the JS
   body's last *two* array lines — the closing `})();` **and** the demo's
   final bootstrap statement (`render();` or `requestAnimationFrame(frame);`).
   The template re-adds only `})();`, so every TS export silently lost its
   bootstrap call → blank canvas (non-RAF) or no animation (RAF). Fix:
   `.slice(K, -2)` → `.slice(K, -1)`.

2. **Independent leading-`K` defect in `iso_saveLoadDemo`.** Its JS body has
   a `// Inject…` comment at line-index 1 that the TS template does not
   mirror. That one-line drift made `.slice(8, …)` resume at JS index 8 =
   `const out = document.getElementById('saveJsonOut');`, which the TS
   template already re-declared with a type cast → duplicate `const out` →
   `Cannot redeclare block-scoped variable 'out'` at runtime. Fix: `K` 8 → 9
   (`iso_mapEditorDemo` has no comment line, so its `K=8` was already
   correct — only Bug 1 applied there).

Never caught because TS exports are Sucrase-gated and the preview sandbox
only validates HTML structure, never executes the TS path (the sandbox
can't even fetch the Sucrase CDN — confirmed during verification).

## Files modified

- `isometric-strategy/bundles-expert.js` — 8 `DEMO_CODE_TS` slice exprs:
  7× `slice(4,-2)`→`slice(4,-1)`, 1× `slice(6,-2)`→`slice(6,-1)` (skirmish).
- `isometric-strategy/bundles-advanced.js` — 8 `DEMO_CODE_TS` slice exprs:
  6× `slice(4,-2)`→`slice(4,-1)`, `iso_saveLoadDemo` `slice(8,-2)`→
  `slice(9,-1)` (both bugs), `iso_mapEditorDemo` `slice(8,-2)`→`slice(8,-1)`.
- No JS (`DEMO_CODE.*`) bodies, templates, or other files touched.
  `bundles-simulations.js` already used the correct `slice(4,-1)`;
  `bundles-beginner.js` / `bundles-intermediate.js` hand-write TS (no slice
  trick) — out of scope, untouched.

## Verification (all passed)

Static server (port 8765, repo root) + browser eval on `expert.html` and
`advanced.html`. For all 16 demos:

- `generateStandaloneHTML(id, deps, 'ts')` embeds the full TS body; its last
  real line == the JS twin's last real line (bootstrap restored).
- `generateStandaloneHTML(id, deps, 'js')` still contains `DEMO_CODE[id]`
  verbatim → JS path byte-unchanged (regression-safe).
- TS body ends with a proper closing IIFE.
- Network-free Sucrase-equivalent strip (`: void`, ` as <Type>`) + `new
  Function()` compiles cleanly for all 16 — no `const` redeclaration.
- `iso_saveLoadDemo`: exactly one `const out`, `render();` in the tail.

## Edge cases considered

- `slice(6)` skirmish (tabindex/focus header) — K=6 correct, only `-2`→`-1`.
- `slice(8)` mapEditor vs saveLoad — diverge because only saveLoad's JS body
  carries a comment line; verified each independently.
- 13 standard `slice(4)` demos — spot-checked headers (fog, combat, stress,
  ai) confirm canvas/ctx/info with blank at index 4; K=4 correct.

## Suggested follow-up (not done)

The preview sandbox never executes the TS export path, which is why a
16-demo defect shipped across multiple commits. Worth adding a build-time
or test-time assertion that each `DEMO_CODE_TS` body's last non-`})();`
line equals its `DEMO_CODE` twin's, so this class of slice drift is caught
automatically.
