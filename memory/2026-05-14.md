## Key technical decisions in this tier

- **A* implementation choice**: 4-connected only (no diagonals), Manhattan
  heuristic, sorted-array open list. Keeps the code ~30 lines and the math
  obvious. Will revisit in Advanced when we need 1000-unit-scale performance —
  there's a key-point callout in `expert.html` flagging that future swap.
- **A* visited-set visualization**: `aStarPath` accepts an optional `visitedOut`
  Set; the pathfinding demo passes one in to render the explored cells as a
  heatmap, making the "why A* is better than BFS" lesson visual.
- **Selection happens in screen space**: the drag-box compares each unit's
  current screen position (via `cartToIso`) against the rectangle. Tile-space
  would work too but screen-space matches what the player sees and is what AoE2
  / RA2 do under the hood.
- **Building placement validity** is a single tight loop: for every cell in
  the footprint, check `isWalkable(map, …)` and a `blockedSet` of currently
  occupied cells. Ghost preview uses semi-transparent fills (green = OK,
  red = blocked) so the rule is felt before it's read.
- **Worker state machine**: 4 states (`walking_to_resource` → `gathering` →
  `walking_to_dropoff` → repeat). Drawn in the gathering demo and reused in the
  skirmish mini-project. Workers in the skirmish ignore combat by design — the
  Advanced tier introduces flee/auto-defense behaviors.
- **Combat**: minimal viable model. HP, attackDamage, attackRange,
  attackCooldown (with reset to attackCooldownMax). Ranged units spawn small
  projectile dots as a visual hint. Range > 1.5 is "ranged"; everything else is
  melee. Future tiers refine with armor types / damage tables.
- **Skirmish mini-project** integrates everything: player base (town centre +
  worker auto-harvesting + a soldier), enemy base across a river that
  auto-produces soldiers, drag-box selection + right-click move (A* around the
  water), and combat on contact. Deliberately leaves out fog of war, group
  collision avoidance, and real enemy AI — those are Advanced tier hooks.

### Key technical decisions in this tier
- **Binary heap A***: graduated from the Expert tier's sorted-array open list
  to a proper `MinHeap`. The heap, the parent-pointer `Int32Array`, and a
  `Uint8Array` visited mark let A* run thousands of times per second on the
  stress-test map. Production-grade scaffolding without the production
  complexity.
- **Flow fields as the swarm primitive**: the 500-unit stress test uses **one**
  flow field followed by **all** units. Each unit's per-frame work is now
  O(1) — read its current tile's flow vector and step. Click-to-set-goal
  rebuilds the BFS cost grid in well under a millisecond for the 28×18 grid.
- **Spatial hash with metering**: the demo runs both the spatial query AND a
  naive O(N) scan on every hover, and reports both timings. At N=300 the
  naive scan is still trivially fast — the point is to show the *technique*
  and let learners feel the bucket structure. The "real" win is at N=5000+,
  noted in prose.
- **Fog of war as `Uint8Array`**: per-cell state is a single byte (0/1/2 for
  hidden/explored/visible). Update cost is O(units × radius²) per frame —
  cheap because both are small.
- **JSON save/load with `version` field**: the save snapshot is an explicit
  allow-list of fields, not a wholesale `JSON.stringify(world)`. Adding the
  `version` from day one — flagged as a callout — saves you the day you
  rename a field.
- **`<details>`-wrapped JSON output panels**: both save/load and map-editor
  expose their JSON inside collapsed `<details>` blocks. This had one verification
  gotcha: `innerText` returns empty for hidden details, so the test harness
  had to switch to `innerHTML` to confirm the export.
- **AI state machine**: simple but complete. Three states (`scout`, `expand`,
  `attack`) with a 1-second slow tick and a baseline economy. Two of these
  AIs running against each other produce credible RTS behavior — bases grow,
  armies form, one attacks, the other counters.

### Architecture takeaway from Phases 1-5
The track now has 4 of 5 tiers shipped end-to-end. The composition pattern
that worked across all four:
1. Track index lists tiers as cards, each with its own status.
2. Each tier HTML follows the same skeleton (head + nav + TOC + sections + scripts).
3. Each tier's demos JS uses IIFEs for isolation + a small set of file-local
   helpers (palette, terrain renderer, etc.) at the top.
4. `shared/utils.js` only gets additions when something is genuinely
   cross-tier (only the 4 iso primitives so far).
5. Verification combines pixel-count canvas checks, simulated DOM events, and
   measurement reads of the info-display strings.

Future tracks (platformer, roguelike, etc.) can clone this scaffolding.
