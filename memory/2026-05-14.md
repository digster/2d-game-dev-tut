## Key technical decisions in this tier

- **A* implementation choice**: 4-connected only (no diagonals), Manhattan
  heuristic, sorted-array open list. Keeps the code ~30 lines and the math
  obvious. Will revisit in Advanced when we need 1000-unit-scale performance —
  there's a key-point callout in `expert.html` flagging that future swap.
- **A* visited-set visualization**: `aStarPath` accepts an optional `visitedOut`
  Set; the pathfinding demo passes one in to render the explored cells as a
  heatmap, making the "why A* is better than BFS" lesson visual.
- **Selection happens in screen space**: the drag-box compares each unit's
  current screen position (via `cartToIso`) against the rectangle. Tile-space
  would work too but screen-space matches what the player sees and is what AoE2
  / RA2 do under the hood.
- **Building placement validity** is a single tight loop: for every cell in
  the footprint, check `isWalkable(map, …)` and a `blockedSet` of currently
  occupied cells. Ghost preview uses semi-transparent fills (green = OK,
  red = blocked) so the rule is felt before it's read.
- **Worker state machine**: 4 states (`walking_to_resource` → `gathering` →
  `walking_to_dropoff` → repeat). Drawn in the gathering demo and reused in the
  skirmish mini-project. Workers in the skirmish ignore combat by design — the
  Advanced tier introduces flee/auto-defense behaviors.
- **Combat**: minimal viable model. HP, attackDamage, attackRange,
  attackCooldown (with reset to attackCooldownMax). Ranged units spawn small
  projectile dots as a visual hint. Range > 1.5 is "ranged"; everything else is
  melee. Future tiers refine with armor types / damage tables.
- **Skirmish mini-project** integrates everything: player base (town centre +
  worker auto-harvesting + a soldier), enemy base across a river that
  auto-produces soldiers, drag-box selection + right-click move (A* around the
  water), and combat on contact. Deliberately leaves out fog of war, group
  collision avoidance, and real enemy AI — those are Advanced tier hooks.

### Key technical decisions in this tier
- **Binary heap A***: graduated from the Expert tier's sorted-array open list
  to a proper `MinHeap`. The heap, the parent-pointer `Int32Array`, and a
  `Uint8Array` visited mark let A* run thousands of times per second on the
  stress-test map. Production-grade scaffolding without the production
  complexity.
- **Flow fields as the swarm primitive**: the 500-unit stress test uses **one**
  flow field followed by **all** units. Each unit's per-frame work is now
  O(1) — read its current tile's flow vector and step. Click-to-set-goal
  rebuilds the BFS cost grid in well under a millisecond for the 28×18 grid.
- **Spatial hash with metering**: the demo runs both the spatial query AND a
  naive O(N) scan on every hover, and reports both timings. At N=300 the
  naive scan is still trivially fast — the point is to show the *technique*
  and let learners feel the bucket structure. The "real" win is at N=5000+,
  noted in prose.
- **Fog of war as `Uint8Array`**: per-cell state is a single byte (0/1/2 for
  hidden/explored/visible). Update cost is O(units × radius²) per frame —
  cheap because both are small.
- **JSON save/load with `version` field**: the save snapshot is an explicit
  allow-list of fields, not a wholesale `JSON.stringify(world)`. Adding the
  `version` from day one — flagged as a callout — saves you the day you
  rename a field.
- **`<details>`-wrapped JSON output panels**: both save/load and map-editor
  expose their JSON inside collapsed `<details>` blocks. This had one verification
  gotcha: `innerText` returns empty for hidden details, so the test harness
  had to switch to `innerHTML` to confirm the export.
- **AI state machine**: simple but complete. Three states (`scout`, `expand`,
  `attack`) with a 1-second slow tick and a baseline economy. Two of these
  AIs running against each other produce credible RTS behavior — bases grow,
  armies form, one attacks, the other counters.

### Architecture takeaway from Phases 1-5
The track now has 4 of 5 tiers shipped end-to-end. The composition pattern
that worked across all four:
1. Track index lists tiers as cards, each with its own status.
2. Each tier HTML follows the same skeleton (head + nav + TOC + sections + scripts).
3. Each tier's demos JS uses IIFEs for isolation + a small set of file-local
   helpers (palette, terrain renderer, etc.) at the top.
4. `shared/utils.js` only gets additions when something is genuinely
   cross-tier (only the 4 iso primitives so far).
5. Verification combines pixel-count canvas checks, simulated DOM events, and
   measurement reads of the info-display strings.

Future tracks (platformer, roguelike, etc.) can clone this scaffolding.

### Key technical decisions in this tier
- **Single-file convention**: matched `simulation.html` / `simulation-v2.html` —
  no `simulations-demos.js`. Each sim is fully self-contained, so a learner
  can copy any one simulation into their own project without untangling shared
  helpers.
- **Value noise instead of true Perlin**: ~25 lines vs ~80, identical visual
  character at the resolutions we care about. Hashing-based gradient hashing
  with smoothstep interpolation.
- **Density-aware continuum simulation**: agents read the BFS flow field at
  their current pixel cell, sample local cell density, and scale their speed
  proportionally — captures the qualitative difference vs boids (which slow
  via repulsion forces) without the full Treuille/Cooper/Popović density
  advection.
- **Page-local CSS**: a single `<style>` block in the head defines `.slider-row`.
  Promoted to `shared/styles.css` would have been premature — only this one
  page uses sliders right now.

### Architecture takeaway from all six phases
The patterns that survived from Phase 1 through Phase 6:
1. **Subdirectory-per-track** with `../shared/` references kept the root clean.
2. **Track-local index page** with a roadmap of tier cards (Coming soon / Ready)
   gave learners scope upfront and us a single source of truth for tier status.
3. **Per-tier files** with the same skeleton (head + nav + TOC + sections +
   scripts) made every tier feel like the same site visually.
4. **IIFE-wrapped demos with a small file-local helpers block** scaled cleanly
   from 7 demos (Beginner) to 8 (Expert / Advanced).
5. **The single-file Simulations convention** turned out to be valuable for
   exactly the reason the original `simulation.html` author chose it: each
   simulation copies cleanly to a learner's own project.
6. **shared/utils.js stayed lean** — only 4 iso primitives added across the
   whole track. Game-specific logic (Unit, Camera, TileMap, A*, flow fields,
   AI state machines, spatial hash, MinHeap) all stayed in the demo files.

is reusable.

  focused; the three new demos already deliver the lesson.

### Three new demos
1. **walkerDemo** — procedurally-drawn 6-frame stick-figure walker. Controls:
   Play/Pause, Step, Reset, FPS slider (1–24), manual frame scrubber (0–5).
   Filmstrip at the bottom shows all 6 frames with the current one highlighted.
2. **stateMachineDemo** — 4-state machine (idle/walk/attack/death) using the
   same walker substrate. Idle = subtle bob, walk = 6-frame cycle, attack =
   3-frame sword swing with a `damage` event on frame 1, death = 4-frame
   collapse (doesn't loop). Damage events use edge-detection so they fire
   *exactly once* per attack cycle even though the renderer runs at 60fps.
   Damage flash renders for 0.5s after each event. Damage counter on the HUD.
3. **easingDemo** — 4-panel side-by-side comparison of linear, easeInCubic,
   easeOutCubic, and easeInOutCubic. Each panel has a moving dot on a track
   and a curve plot showing the easing function. Slower/Faster buttons adjust
   duration in 0.5s steps. The dashed diagonal reference line in each plot
   visualizes how the curve differs from identity.

### Key technical decisions in this addition
- **Procedurally-drawn frames** (limb angles as numbers) instead of sprite
  sheets. Sidesteps the Phase 2 "no external assets" decision while still
  teaching the actual playhead mechanism. Frame data is just objects like
  `{ lLeg: 0.6, rLeg: -0.6, lArm: -0.5, rArm: 0.5 }` — the renderer reads
  those angles each frame.
- **Edge-detected per-frame events** (`if (idx !== lastFrameIdx)`). Without
  this check, a `damage` event scheduled for frame 1 would fire ~8 times per
  attack cycle (once per render frame during the ~167ms the animation spends
  on frame 1 at 6fps). Edge detection makes "deal damage on frame N" actually
  mean "exactly once".
- **Auto-return to idle after non-looping clips finish**. The attack state
  machine returns to idle when the attack clip's `done` flag flips. Death
  doesn't auto-return because death is a terminal state; reset button revives.
- **TOC entry placed between `#eight-dir` and `#mini-project`** — the new
  section references the 8-direction logic ("facing picks the row, animation
  picks the column") so it sits naturally after it.
- **`.slider-row` CSS** was duplicated inline rather than promoted to
  `shared/styles.css`. Two pages use it now (simulations + intermediate). Rule:
  promote when a *third* page picks it up.

### Architecture note
After this addition, the tier teaches three orthogonal "interpolation over time"
concepts as separate sections, then composes them in the mini-project:
- `#units` — position over time (linear lerp between tile coords).
- `#eight-dir` — angle → row index selection.
- `#animation` — visual frame over time (playhead through clip frames).

The composition rule is now explicit in the new section's intro: facing picks
the row, animation picks the column, motion picks the position. Three
independent interpolations.
