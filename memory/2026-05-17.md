## Scope deviation (intentional, flagged to user)

Plan listed *slime mold (Physarum)* for Simulations; **swapped for the 2D
wave equation**. `makeSim` is single-buffer ping-pong; Physarum needs agents
that *sense* a separately-diffused trail map (two coupled fields the harness
can't feed back). Wave fits the single-state model perfectly (like
reaction-diffusion). Optional 4th lessons per tier were **not** built (kept
to the verified core set).

## Key learnings / gotchas

- **`node -c <file>` every edited bundle/demo BEFORE browser testing.** A
  closing `DEMO_CODE` delimiter was over-escaped (`})();\`;` instead of
  `` })();`; ``) in bundles-intermediate.js → template literal never
  terminated, bled into the next block → "Unexpected token ':'". Caught
  instantly by `node -c`; the inner GLSL backticks ARE escaped, the final
  one is NOT.
- **Test oracle:** `makeShaderToy`/`makeSim` `fail()` flips the info element
  to `#ff7b72` on any GLSL compile/link error. Asserting computed color
  stays grey (`rgb(158,158,158)` / `rgb(139,148,158)` standalone) = "shader
  compiled" without pixel readback.
- **Export verification:** `generateStandaloneHTML(id, deps, lang)` is a
  global; call it in-page, save via Playwright `filename`, decode the
  (double-)JSON-wrapped result, run the standalone, assert 0 console errors.
  Did this for the riskiest demo in every tier. TS export path can't be run
  here (Sucrase CDN blocked in sandbox — pre-existing for ALL repo exports);
  proved TS bundles correct by showing their GLSL is byte-identical to the
  JS twins (which run clean).
- **Advanced DRY contract:** neutral `TEX`/`OUT` bodies in demos.js
  (`g1`/`g2` add PHEAD + resolve dialect) ↔ `kitSrc()` regenerates the same
  via `__VER__/__SDECL__/OUT/TEX` substitution. Inject mode variants with
  `' + x + '` concat (like `gauss`), never `${}`, and keep injected GLSL
  single-line to avoid `\n`→`\\n` escaping in the exported kit.

## Known issue (pre-existing, flagged — NOT introduced as logic bug)

# 2026-05-17 (part 2) — Regression fix: WebGL context exhaustion

## The regression

User reported the FIRST Advanced demo (`advanced.html`, canvas `rttGL1`,
Render-to-Texture) renders blank black. Root cause: every shaders tier page
creates one WebGL context per demo canvas, **eagerly at load, never released**.
`advanced.html` = 9 effects × {GL1 pane, GL2 pane} = **18 contexts** (the GL2
pane is `hidden` but its canvas still exists → `mountFX` still
`getContext`s it). Browsers cap simultaneous WebGL contexts at ~16 and
silently EVICT THE OLDEST → `rttGL1`. `makeFXChain`'s only context-loss
handling was `e.preventDefault()` with no recovery → permanent blank. The
part-1 expansion added the 6 contexts (3 effects × GL1/GL2) that tipped it
over. (Pre-fix counts: Beginner 11, Intermediate 11, Sim 12, Expert 14,
Advanced 18.) User chose: robust fix (lazy + teardown), **all five tiers**.

## The fix

**`shared/lazy-demo.js` (NEW)** — `lazyToy(canvas, factory)`:
- Stable **proxy** mirroring the harness API
  (setFrag/setParam/setPaused/setTimeScale/rebuild/rebuildChain/reset/stop/
  destroy). Button handlers attach ONCE to the proxy; the last call to each
  mutator is recorded and **replayed** after a re-mount so visual state
  survives scroll-away/back.
- **IntersectionObserver** (`rootMargin: '300px 0px'`): on screen → mount
  (`factory(currentCanvas)`); off screen → unmount (`destroy()` the harness).
- **Canvas swap** (the key gotcha): a `<canvas>` is bound to ONE context for
  its lifetime — after `WEBGL_lose_context.loseContext()`, `getContext()` on
  the same canvas keeps returning the dead one. So unmount also
  `replaceChild`s the canvas with a fresh `cloneNode(false)` (id/size/class
  preserved) and re-`observe`s it, so the next mount gets a brand-new context.
  Factory receives the *current* canvas (`cv`), must not close over the outer
  one. *(First attempt without canvas-swap failed exactly here — caught by the
  scroll-back stress test: rtt came back context-lost + red.)*
- No `IntersectionObserver` → mount eagerly, never unmount (old behaviour).

**`destroy()` added** to `makeFXChain` (advanced), all three `makeShaderToy`
copies (beginner/intermediate/expert; expert also `deleteTexture(spriteTex)`),
and `makeSim` (also removes the window-level `mouseup`): cancel rAF, remove
named listeners, delete programs/buffers/textures/FBOs, then
`WEBGL_lose_context.loseContext()`. `stop()` kept for back-compat.

**Wrapped every demo call site** with `lazyToy(canvas, (cv) => makeX(cv, …))`
(beginner ×10, intermediate ×10, expert ×10, simulations ×9, advanced inside
`mountFX` — `rebuildChain` moved into the factory). Added
`<script src="../shared/lazy-demo.js">` before each tier's `*-demos.js`.

## Bundles / Exports — deliberately UNCHANGED

`git status` confirms no `bundles-*.js` / `export-demo.js` changes. Standalone
exports inline only dep bundles (not lazy-demo.js) and have one
always-visible canvas → one context, eager, fine. Spot-checked `sh_sand`
export: byte-identical to part-1 (13879 b), no `lazyToy` inlined, runs 0
errors.

## Accepted trade-off

Stateful demos (sims, ping-pong/feedback) **re-seed when scrolled fully
off-screen and back** (context torn down → re-created). Acceptable for a
tutorial and only when not visible — far better than a blank/evicted canvas.

## Verification (real browser, Playwright)

Per tier: load → top demo's context **not** lost, info grey; walk the ENTIRE
page in 500px steps then back to top; assert **0 errors / 0 warnings**
(crucially **no "Too many active WebGL contexts"** even after every canvas
passed through the viewport — proves teardown frees contexts); top demo
re-mounts and renders; new-lesson controls work post-remount (proxy/replay).
Advanced also: screenshot confirms `rttGL1` renders the 3-blob scene;
WebGL1↔WebGL2 api-tab switch mounts the shown pane / unmounts the hidden one
(no api-tab handler change needed — `hidden` toggles IO). All 6 edited JS
files + `shared/lazy-demo.js` pass `node -c`.

## Files touched (part 2)

simulations}-demos.js` (destroy() + wraps); `shaders/{...}.html` ×5 (script

# 2026-05-17 (part 3) — Resolving the slime-mold scope flag: `makeAgentSim`

## Request

User pointed at the part-1 "scope deviation" flag (wave shipped instead of
slime mold) and asked: explain *why* `makeSim` couldn't do Physarum, then
"how can we set up the project to support examples like slime mould in the
future while still supporting the current example infrastructure?" Chosen via
plan: **sibling harness** (no `makeSim` refactor — zero regression risk) +
**ship the Physarum demo as the first consumer** + **document the seam**.

## The real insight

The project already had the right extensibility seam — it just wasn't named
one. `lazyToy(canvas, factory)` is **harness-agnostic**: it only needs a
factory returning `{ destroy, stop, setParam?, setPaused?, reset?, rebuild? }`.
`makeShaderToy`/`makeFXChain`/`makeSim` all already conform. So supporting
Physarum = add a *fourth* harness that conforms; touch nothing else.

## What shipped

- **`makeAgentSim`** (new, beside `makeSim` in `simulations-demos.js`) — two
  RGBA16F ping-pong pairs (agents `256²` NEAREST/CLAMP; trail screen-sized
  LINEAR/REPEAT = toroidal). Per substep: ① agentStep sense+steer+move
  (gather) ② trailStep 3×3 blur×0.90 decay (gather) ③ **deposit = scatter**:
  attribute-less `gl.drawArrays(POINTS)`, `gl_VertexID`→agent texel→clip pos,
  additive `blendFunc(ONE,ONE)` *on top of* the diffused trail (no clear) ④
  display. Same return shape as `makeSim` → drops into `lazyToy`. `destroy()`
  frees 4 tex + 4 FBO + 6 prog + buffer + window mouseup + loseContext.
- New GLSL kit consts `AGENT_HEAD/TRAIL_HEAD/AGENT_LIB/DEPOSIT_VERT/FRAG`.
- **Physarum demo IIFE** `slime()` + section `#slime` in `simulations.html`
  (concept-box with an ASCII agent↔trail loop diagram + collapsible code),
  TOC entry, `shaders/index.html` topic bullet. Controls Explore/Network/
  Tight map `u_param`→sensor angle `0.30+0.55*u_param`.
- `ARCHITECTURE.md` (NEW — none existed) documents the harness-contract seam,
  the 4 harnesses, "new class ⇒ new harness; new example ⇒ new shaders only".

## Decisions / gotchas

- **`EXT_float_blend` not needed** — it only gates 32-bit float blending;
  RGBA16F additive blend is covered by the `EXT_color_buffer_float` makeSim
  already requires. Verified: deposit blends fine.
- **Standalone Export deliberately deferred.** `export-demo.js` injects an
  Export button into every `<details data-demo-id>` and `console.error`s if
  the id isn't in `bundles-simulations.js`. The bundle is a *hand-maintained,
  lazy-free verbatim copy* of each harness (note live `pass`→bundle `passQ`
  drift) — duplicating a ~180-line harness as a brittle string is the single
  biggest bug risk here. Chose to ship the slime `<details>` **without**
  `data-demo-id` (no broken button, page demo fully works) and flag
  `sh_makeAgentSim`/`sh_slime` registration as a clean follow-up. Recorded in
  ARCHITECTURE.md.
- **Cold start looks like uniform speckle for ~10–15 s** then snaps into the
  classic reticulated vein network — expected Physarum behaviour, not a bug
  (verified across screenshots).

## Verification (real browser, Claude preview MCP)

`python3 -m http.server` (`.claude/launch.json` "static"). Slime: forms the
vein network, stable (no NaN/white-out), **0 console logs/errors** across all
interactions. `Tight` preset visibly re-topologised the net + info text
updated (proves `setParam` via the lazyToy proxy). Scrolled away→back:
slime's context freed and it re-seeded/re-mounted; **Reaction-Diffusion and
Wave (`makeSim` demos) re-mounted and render correctly** — no regression, no
"too many contexts". No screenshot files written to the repo.
