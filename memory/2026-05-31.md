## Architecture notes (so future sessions don't relearn)

- **Real-time, not turn-based.** Unlike the roguelike (one step per keypress), the
  platformer runs a continuous fixed-timestep loop. Hence `pfLoop` (accumulator) +
  held-key input with edge detection, vs the roguelike's `rlInstallCanvasKeys`
  one-action-per-key model.
- **Engine split (the "promote on 2nd/3rd consumer" rule):** truly cross-tier code
  (tilemap, AABB collision, input/loop) lives in `platformer/engine/` from the
  scaffold. *Taught* tier-specific algorithms (swept-slope collision, one-way
  drop-through, the camera, the feel-kit) stay **inline** in their tier's demos
  file until a 2nd consumer triggers promotion — exactly how roguelike grew
  actors.js / dungeon.js / vision.js / rpg.js. Promotion candidates flagged in the
  plan: `engine/camera.js` (Expert + Sim) and `engine/player.js` (Advanced + Expert + Sim).
- **Per-axis collision is THE primitive.** Move X & snap, then move Y & snap using
  the corrected X — keeps "walk into a wall" and "land on a floor" independent so a
  character doesn't catch on the seams between floor tiles.
- **No top-level shadowing of shared/utils.js globals.** Verified utils.js owns:
  Vector2D, Matrix2D, lerp, clamp, map, randomInt, randomFloat, clearCanvas,
  drawGrid, drawVector, lineIntersection, lineIntersectsRect, createDemoController,
  cartToIso, isoToCart, drawIsoTile, pickTileFromMouse. The engine names (PFTile,
  PF, TileMap, drawTileMap, AABB, moveAndCollide, pfInstallKeys, pfLoop, PF_MAX_DT,
  PF_KEYMAP, PF_EPS) collide with none.
- **Export opt-out:** demos will be keyboard/canvas-driven; export-demo.js only
  renders button controls, so tiers omit `data-demo-id` (no bundles-*.js for now).
  `pf_` namespace reserved for future per-tier bundles.
