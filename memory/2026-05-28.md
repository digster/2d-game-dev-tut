## Key conventions locked in (inherit for all five future tier commits)

- **Helpers in `netcode/net/`, not `shared/`.** They're netcode-specific
  for now; promoting to `shared/` would imply other tracks should
  depend on them. ARCHITECTURE.md was updated with the general rule:
  helpers used by ≥ 2 tier files in the same track go in a sibling
  subfolder; helpers used across tracks go in `shared/`.
- **`FakeNetwork` endpoint surface mirrors WebSocket/DataChannel**
  (`send(targetId, msg)`, `onMessage(handler)`) so demo code is
  transport-agnostic. If a real backend ever lands later, the demos
  swap in O(1) lines.
- **Reorder is implemented as "occasionally add extra delay"** rather
  than as a queue swap. Same observable effect (later message arrives
  first), simpler code, composes cleanly with jitter — a reordered
  packet is just one with an unusually large jitter sample.
- **`SeededRng` is the only randomness inside `FakeNetwork`.** No
  `Math.random()`. The same (seed, message sequence) tuple ALWAYS
  produces the same stats — proven in stress test #3 above.
- **`mulberry32` reaffirmed as the project's PRNG of choice.**
  Racing-sim's procgen uses it; netcode uses it; future tracks should
  too. Constructor nudges `seed=0` to `1` to dodge the known degenerate
  zero-sequence.
- **The `<details>` collapsible code blocks intentionally absent from
  the landing page** because no tier files ship yet (no code to show).
  The Beginner tier commit will introduce them. `export-demo.js` is
  not loaded yet for the same reason; once tier files exist, each one
  will register its demos in a `bundles-<tier>.js` (or omit
  `data-demo-id` to opt out, same approach racing-sim used in its
  first commit).

## The four Beginner-tier demos (`net_` namespace reserved for future bundles)

1. **packetLaneDemo** — animated single-direction lane CLIENT → SERVER.
   Auto-sends a packet every 700 ms. Each in-flight packet drawn as a
   labelled circle at progress = (now − sentAt) / delay (which is why
   the FakeNetwork additive tweak was needed). Sliders for RTT
   (0–500 ms), jitter (0–100 ms), loss (0–50%), reorder (0–50%).
   Lost packets shown as a faded red ✕ marker that scrolls and fades
   over 800 ms. Reordered packets tinted purple with a fat halo.
   Pause + reset-stats buttons. Stats line shows
   sent / delivered (% drop) / reordered (%). This is the demo that
   proves the FakeNetwork actually behaves as advertised; everything
   later in the tier rests on the user having seen this one.

2. **tickVsFrameDemo** — TWO horizontal rails: ticks on top, render
   frames on the bottom. Bar moves at constant velocity, bounces at
   edges. Orange SIM ball on the tick rail snaps to each new tick
   position; cyan REN ball on the render rail tracks SIM either by
   snap (latest tick value) or by lerp(prev, curr, tickAccum /
   tickPeriod), toggle-controlled. Slider for tick rate (2–60 Hz).
   The dashed yellow connector between SIM and REN shows the visible
   gap (also reported numerically in the info bar as
   `gap |sim−ren| = N px`). Plants the seed the Intermediate tier's
   entity interpolation will pay off — they're the same technique,
   one on local data and one on data arriving from another machine.

3. **bandwidthCalcDemo** — DOM-driven (no canvas). Four sliders:
   tick rate (1–64 Hz), players (2–100), entities/player (1–50),
   bytes/entity (2–128). Two readout rows: per-client outbound, server
   total upstream. Coloured severity meter (modem / mobile-DSL /
   broadband / "too much") whose fill width is `log10(kbps) /
   log10(10000)` mapped to 0–100% — gives logarithmic feel across the
   four orders of magnitude the equation can produce. Four shipped-game
   presets (Quake-era 10 Hz / 16 players, Valorant-ish 64-tick,
   battle royale 64 players, MMO 2 Hz / many entities). The verdict
   text below the meter calls back to the Expert and Advanced tiers'
   work (AoI for `playerCount`, delta compression + quantization for
   `bytesPerEntity`).

4. **pingPongDemo** (capstone mini-project) — two clients A and B,
   each rendered as a coloured panel inside the canvas, exchanging
   messages through one `FakeNetwork` in the middle. Two-way lane:
   A→B packets drawn on the upper half, B→A on the lower. Buttons
   under each client: "Send ping" (one) and "Send burst (5)" (five
   in quick succession — exposes jitter visibly since 5 sends at
   ~same time get spread out by jitter). Seed input + Reset button
   rebuilds the network with the chosen seed. Per-client stats
   (sent/received counters and a log of the last 5 receives, late
   arrivals tagged `[late]` in purple) plus a global network stats
   line under the canvas. The intentional demonstration: set
   seed=42, click burst A then burst B, write down the global stats;
   reset, repeat → stats land in the exact same place every time.
   That's the substrate the Expert tier's lockstep and rollback
   stories are built on, and the Beginner mini-project proves it
   by hand.

## Key conventions locked in (inherit for all four future tier commits)

- **Colour palette.** `NET_COLORS` is the shared visual family for the
  whole netcode track. Cyan = client, orange = server, green =
  delivered, red = dropped, purple = reordered. Memorised on day one,
  reused in every later tier so the visual language stays consistent
  across all 20+ demos the track will eventually have.
- **Endpoint surface mirrors WebSocket/DataChannel.** Every demo
  uses `endpoint.send(targetId, msg)` and
  `endpoint.onMessage(handler)` — the exact two-method surface a
  real transport would expose. Demo code would survive a swap to a
  real backend transport with minimal change.
- **`MAX_DT = 0.05` clamp** in every per-frame loop. Same value the
  racing-sim track uses. Tab refocus or hot GPU never produces a
  > 50 ms dt that could break animations.
- **FakeNetwork inspection-by-property is intentional.** Demos read
  `net.inFlight` (the sorted in-flight queue) and `net.stats` (the
  counters) directly to render visualisations. Both are public
  properties, not `_`-prefixed — the visualisation use case is a
  first-class user of the helper, not a "reaching into internals"
  smell.
- **Loss/reorder fate detection by stats delta.** When a demo calls
  `endpoint.send(...)` it snapshots `net.stats.dropped` /
  `net.stats.reordered` BEFORE the call and compares AFTER to
  decide what happened to that specific packet. This is more robust
  than wrapping the FakeNetwork internals; it relies only on the
  public stats counters and survives any future change to the
  enqueue strategy.
- **Top-level safe names in `beginner-demos.js`:** `NET_COLORS`,
  `MAX_DT`, `fmtMs`, `fmtKbps`, `fmtBytes`, `formatNumber`. None
  collide with `shared/utils.js` exports — verified with
  `grep -E "^(class|function|const|let|var)" shared/utils.js`
  before write.
- **`<details>` code blocks intentionally OMIT `data-demo-id`** because
  no `netcode/bundles-beginner.js` ships yet. Same opt-out approach the
  racing-sim Beginner tier used to keep the Export buttons silent until

## Edge cases & test ideas surfaced this iteration

- **Burst send + jitter shuffles arrival order even at 0% reorder rate.**
  Five packets sent within ~1 ms can easily land out of order under
  ±20 ms jitter. This is realistic — Internet jitter alone is enough to
  reorder UDP without any "reorder" event class. Worth surfacing as a
  teaching note in the Intermediate tier when entity interpolation has
  to handle out-of-order snapshots.
- **`updateHud` runs on RAF, not immediately after `.click()`.** The
  ping-pong stats counters increment synchronously inside `send()`, but
  the DOM doesn't update until the next animation frame. Programmatic
  tests need to await a frame before reading the DOM. Not a bug;
  documented in the determinism test code.
- **Bandwidth meter saturates at 10 Mbps.** The log-scale cap is fine
  for teaching but a user pushing all sliders to max can reach ~16 Mbps
  and the meter just stays at "TOO MUCH". Could refine the cap or add
  a second "ludicrous" band in a follow-up.

5. **arenaDemo** (capstone) — SERVER + CLIENT panels. ONE local

## Key conventions locked in (inherit for the next three tiers)

- **Server-and-client state lives in SEPARATE objects** in every demo.
  Communication is ONLY through `endpoint.send` / `endpoint.onMessage`.
  Discipline note typed verbatim into the file header: "if a demo
  reads `serverPlayer.position` from a client renderer, it's lying
  about what the client actually knows". This is the critical
  discipline the rest of the track depends on — the moment a demo
  cheats by reading server state directly, the technique it's
  supposed to teach becomes invisible.
- **Top-level safe names in `intermediate-demos.js`:** `NET_COLORS`
  (identical to beginner-demos.js — same palette, no collision
  because each tier page loads only its own *-demos.js), `MAX_DT`,
  `SIM_HZ` (60), `INPUT_TICK_HZ` (30), `WORLD_W` (320), `WORLD_H`
  (220), `SPEED` (140), `Player`, `applyInput`, `integrate`,
  `cloneSnapshot`, `lerpV`, `fmtMs`, `drawPanel`, `drawPlayer`,
  `drawGhost`, `attachKeyboardInput`, `attachButtonInput`. Pre-greped
  against `shared/utils.js` — none collide.
- **Visual convention: SERVER orange (LEFT) | CLIENT cyan (RIGHT).**
  Every two-panel demo follows this layout. Green = predicted /
  interpolated / smoothed overlay. Purple = remote players. Red =
  raw correction snaps. The whole track will keep these.
- **Frame-rate-correct exponential decay** `1 − exp(−k·dt)` where
  `k = −ln(0.01) / (smoothMs/1000)` for "99% of the gap closed in
  smoothMs". This is the same formula racing-sim/expert's follow
  camera uses; the smoothing code block in `intermediate.html`
  links to it explicitly so learners see the technique transfers
  across domains.
- **Server tick rate vs snapshot rate vs input rate are SEPARATE
  numbers.** SIM_HZ=60 (server's internal sim), SNAPSHOT_INTERVAL=
  1/15 or 1/20 depending on demo (network broadcast), INPUT_TICK_HZ=
  30 (client's input upload). The Beginner tier's tick-vs-frame
  demo established this; the Intermediate tier uses all three.
- **The on-screen directional pad is the primary input** for demos
  where keyboard would require canvas focus. Each pad button has
  mousedown/mouseup/mouseleave/touchstart/touchend bindings so it
  Just Works on desktop and mobile. WASD/arrows on a focused canvas
  still work as a secondary path.

## Edge cases & test ideas surfaced this iteration

- **Synthetic `MouseEvent('mousedown')` events don't perfectly
  reproduce a real user hold.** During verification, dispatching
  mousedown/mouseup on the direction buttons via JS sometimes
  produced different end positions than the analytical expectation.
  Real user touches/clicks work correctly (visual confirmation).
  Worth a note for any future automated browser tests:
  use real input dispatch or measure end-state rather than
  mid-transit positions.
- **Pixel-sampling-against-panel-borders is fragile.** Every panel
  is drawn with a coloured stroke (NET_COLORS.server/client), and
  the border edges anti-alias. A naive "find orange pixels in this
  row" detector matches border AA across the whole panel width.
  Lesson for future verification: skip borders, sample only the
  INTERIOR (e.g. x in [oxS+3, oxS+WORLD_W-3]), or take a screenshot.
- **Client clock-offset estimation** in `interpolationDemo` and
  `arenaDemo` is naïve — set once on first snapshot arrival, never
  re-estimated. In real netcode you'd use a moving filter (EMA of
  observed server-time vs local-time deltas). The simplified
  version is fine for teaching because it stays correct AS LONG AS
  the user doesn't open the tab, sleep the machine, and come back
  — in which case the offset would drift. Worth a one-paragraph
  note in the Advanced tier's reconciliation section.
- **`mouseleave` on the directional pad cancels the input.** If
  the user drags off a held button instead of releasing on it, the
  input cancels. This is the correct mobile-friendly behaviour but
  could feel "sticky" with very fast finger drags. Default desktop
  behaviour matches every other web UI.

- **Negative HP death-spiral in shooterDemo.** First iteration had no
  respawn logic — `sLocal.health--` would just keep decrementing forever

  bump the version param** (or we could switch to a stronger
  cache-busting strategy if this becomes painful). For real-user
  visits this isn't a problem (cold cache + Last-Modified validation
