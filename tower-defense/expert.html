♻️ Object pooling — reuse, don't allocate

Spawning thousands of short-lived objects (sparks, shots) hands the garbage collector a pile of work, and a GC pause is a dropped frame. A pool recycles instead: dead objects go on a free list and the next spawn reuses one, allocating only when the list runs dry. Watch the counter: in naive mode "objects ever allocated" climbs forever; pooled, it plateaus once the pool is warm.

📝 Show the code: a free-list pool 
class TDPool {
  spawn(init) {
    let o = this.free.pop();                       // reuse a recycled object…
    if (!o) { o = this.factory(); this.allocs++; } // …allocate ONLY when starved
    init(o); this.live.push(o); return o;
  }
  sweep() { /* dead objects → the free list (O(1) swap-remove), never the GC */ }
}

🧮 Struct-of-Arrays — memory layout matters

An Array-of-Structs ([{x,y,vx,vy}, …]) scatters each creep across the heap — iterating chases a pointer per creep. A Struct-of-Arrays stores one flat Float32Array per field, so all the x's sit contiguous in memory: the loop becomes a linear walk over packed data, with no boxing and nothing for the GC to collect. Both stores run the same N agents below; compare the update time (the gap widens with more fields and bigger N — and SoA is what makes typed-array stores like the swarm's possible at all).

📝 Show the code: SoA + swap-remove 
// Array-of-Structs: a heap object (and a pointer chase) per creep
const aos = [{ x, y, vx, vy }, /* … */];

// Struct-of-Arrays: one flat typed array per field — packed, GC-free, cache-friendly
const x = new Float32Array(N), y = new Float32Array(N);
for (let i = 0; i < N; i++) { x[i] += vx[i] * dt; y[i] += vy[i] * dt; }

// death is O(1) swap-remove — copy the last creep over the gap, shrink the count
function remove(i) { const last = --count; x[i] = x[last]; y[i] = y[last]; }

🗂️ Spatial hashing — range queries without the blow-up

"Which creeps can this tower hit?" asked naively is every tower against every creep — O(towers × creeps), which explodes. A spatial hash buckets each creep into a grid cell once per frame; a tower then reads only the handful of cells its range overlaps. Pick the cell size ≈ the range and a query touches a small, constant patch. The counter shows the pair-tests saved (the highlighted cells are the first tower's query).

📝 Show the code: bucket & query 
// bucket every creep once (cell size ≈ the largest query range)
hash.clear();
for (let i = 0; i < creeps.length; i++) hash.insert(i, creeps[i].x, creeps[i].y);

// a tower reads only the cells its range overlaps → ~k candidates, not E
const near = hash.queryInto(tower.x, tower.y, tower.range, out);
for (const i of near) { /* exact distance check on the few that matter */ }

🖌️ Render batching — one path for the crowd

Drawing each creep with its own beginPath/fill means thousands of separate draw calls — most of the cost is the calls, not the pixels. Batch them: one beginPath, add every creep's arc with moveTo+arc, then a single fill. Toggle the two and watch the render time.

📝 Show the code: one path, one fill 
// SLOW — a beginPath + fill per dot (thousands of draw calls)
for (const d of dots) { ctx.beginPath(); ctx.arc(d.x, d.y, 2, 0, TAU); ctx.fill(); }

// FAST — one path for the whole crowd, one fill (same colour)
ctx.fillStyle = colour; ctx.beginPath();
for (const d of dots) { ctx.moveTo(d.x + 2, d.y); ctx.arc(d.x, d.y, 2, 0, TAU); }
ctx.fill();

🌊 Scale — thousands on one flow field

Put SoA storage and the engine's tdFlowField together: thousands of creeps, each a row in a Float32Array, all steering by one field computed once. Adding a creep costs a single array lookup per step — so the steer time barely moves as the crowd grows. Crank the spawn rate; the pillars split the stream.
