🧱 What a tower defense really is

Strip away the art and a tower defense is a tug-of-war between two simple jobs, sixty times a second:

  • The attacker spawns creeps that walk a path from a spawn point to your goal. Each creep is just a position, some HP, and a speed.
  • The defender (you) places towers on the map. A tower has a range, picks a target inside it, and fires on a cooldown.
  • The stakes — a creep that reaches the goal costs you a life; a creep you kill pays gold; gold buys more towers.

That's the whole skeleton, and we build it one piece at a time on the engine the track scaffold already gave us: TDGrid (the tiles), TDPath (the lane), tdLoop (the fixed-timestep clock), and the tdDraw* helpers (the look). Everything later in the track — many tower types, enemies that re-route around your towers, thousands of creeps — is more of this same loop with cleverer rules.

💡 Two questions run the whole genre. The defender asks "what do I hit, and where will it be?" — target choice plus aiming. The attacker asks "what's the cheapest route to the goal?" — which becomes A* and flow fields in the Advanced tier. This Beginner tier answers the defender's first half (hit the creep furthest along) and the attacker's simplest case (one fixed lane).
⚠️ One trap worth knowing up front: our Vector2D (from the shared utils) mutates in place — v.add(...), v.multiply(...), v.normalize() change v; only v.subtract(...) and v.copy() return a new vector. An entity owns its own pos, so mutating that is fine — but to get a "toward the target" direction we use target.pos.subtract(myPos), which returns a fresh vector and leaves the target's position untouched.

🗺️ The map — a grid, a lane, and where you can build

The map is a tile grid (TDGrid). Two things live on it: a lane the creeps walk (drawn from a TDPath), and the towers you place. A tower can go on any tile that is in-bounds, not on the lane, and not already taken — that's the single question grid.isBuildable(col, row) answers. Hover the map below: green tiles are buildable, the red corridor is the lane.

Toggle the lane between straight (the raw waypoints) and smooth — the same TDPath can round its corners with a Catmull-Rom curve, a preview of the curved lanes the Intermediate tier builds.

📝 Show the code: the map & buildability 
const grid = new TDGrid(19, 9, 40);        // 19×9 tiles of 40px each
const path = new TDPath(waypoints);        // a lane through [{x,y}, …]
grid.blockAlongPath(path);                 // stamp the lane as un-buildable

// the one question tower placement asks:
grid.isBuildable(col, row);   // in-bounds, not lane, not already occupied?

// pixel → cell, and cell → its centre (where a tower sits)
const cell = grid.worldToCell(mouseX, mouseY);
const { x, y } = grid.cellCenter(cell.col, cell.row);

🐛 The creep — following a path is one moving number

Here's the trick that makes path-following trivial: a creep doesn't store a list of waypoints to chase. It stores a single number — dist, how far along the lane it has travelled — and each step does dist += speed × dt. The lane is arc-length parameterized, so it can turn that one number back into an (x, y) with path.pointAt(dist).

One scalar slides from 0 to the lane's length; pointAt maps it onto the curve. Speed is in px/s, so the creep moves the same real distance whatever the frame rate.

📝 Show the code: TDEnemy.update 
class TDEnemy {
  update(dt) {
    this.dist += this.speed * dt;             // advance along the lane
    const p = this.path.pointAt(this.dist);   // arc-length → (x, y)
    this.pos.set(p.x, p.y);
    if (this.dist >= this.path.length) {      // reached the goal…
      this.leaked = true;                     // …and "leaked" (costs a life)
      this.alive = false;
    }
  }
}

🌊 The spawner — a wave is a countdown on a cadence

One creep isn't a game; a wave is. And a spawner is the simplest possible machine: a counter of how many are left, and a timer. While creeps remain, every interval seconds it emits one and ticks the counter down. That's it — change the count and the interval and you've changed the wave's size and density.

📝 Show the code: the spawner 
// a wave is just a countdown that emits on a cadence
if (spawner.remaining > 0) {
  spawner.timer -= dt;
  if (spawner.timer <= 0) {
    spawner.timer = spawner.interval;          // reset the cadence
    spawner.remaining--;
    creeps.push(new TDEnemy(path, { speed }));
  }
}

🔫 The tower — range, target acquisition & fire-rate

A tower does three things every step: keep or find a target, aim at it, and — when its cooldown expires — fire. Two ideas matter here. First, range is a circle, and "is this creep in range?" is a distance check we do with squared distances (no Math.sqrt — comparing against the squared radius gives the same answer for less work). Second, fire-rate is a cooldown: a tower that fires twice a second resets a 1/2 = 0.5 s timer after each shot.

Which creep does it pick? The classic default — the one furthest along the lane (closest to the goal), the most urgent threat. In this demo each shot is an instant hitscan beam so you can focus on range and rate; the next section gives the shot a body. Click a green tile to move the tower and feel how range interacts with the lane's bends.

📝 Show the code: range, acquisition & cooldown 
// in range? compare SQUARED distances (skip the sqrt)
inRange(e) { return this.pos.distanceSquared(e.pos) <= this.range * this.range; }

// the "first" targeting mode: of the creeps in range, the one furthest along
function tdPickTarget(tower, enemies) {
  let best = null, bestProgress = -Infinity;
  for (const e of enemies) {
    if (e.alive && tower.inRange(e) && e.dist > bestProgress) {
      bestProgress = e.dist; best = e;
    }
  }
  return best;
}

// fire-rate is a cooldown of 1 / fireRate seconds
this.cooldown -= dt;
if (this.cooldown <= 0) { this.cooldown = 1 / this.fireRate; /* FIRE */ }

💥 The projectile — giving the shot a body

A hitscan beam hits instantly. A projectile is a moving point that chases the target's current position at a finite speed — which makes the game feel physical and introduces two honest consequences. A slow shot can be outrun by a fast creep (watch the orange ones), and if the target dies mid-flight the shot fizzles (overkill is real). The first problem is exactly what the Intermediate tier's lead-the-target math fixes — aiming where the creep will be.

The hit test uses the sub-step reach (speed × dt + targetRadius): checking "did I get close enough this step?" rather than "am I exactly on it?" so a fast shot can't tunnel straight through a creep between frames.

📝 Show the code: TDProjectile.update 
update(dt) {
  if (!this.target || !this.target.alive) { this.alive = false; return; } // fizzle

  // a FRESH vector toward the target (subtract returns new — no mutation)
  const toward = this.target.pos.subtract(this.pos);
  const reach = this.speed * dt + this.target.radius;   // sub-step hit window
  if (toward.lengthSquared() <= reach * reach) {        // close enough → hit
    this.target.damage(this.damage);
    this.alive = false;
  } else {
    this.pos.add(toward.normalize().multiply(this.speed * dt)); // step toward it
  }
}
