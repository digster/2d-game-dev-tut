🧱 What a platformer really is

Strip away the art and a 2D platformer is three things working together, sixty times a second:

  • A loop that advances the world by a tiny, fixed slice of time every step — read input, move things, draw.
  • A box (the player) with a position and a velocity, pulled down by gravity.
  • A grid of solid tiles (the level) that the box is not allowed to pass through.

That's the whole skeleton. Coyote time, wall-jumps, dashes, cameras — everything later in this track is a refinement of these three. So we build them first, in order, each demo adding exactly one idea.

💡 Why a fixed timestep? If you move the player by velocity × (time since last frame), the game behaves differently on a 30 Hz laptop and a 240 Hz monitor — and physics can even glitch when a frame hitches. Instead we advance the simulation in fixed 1/60-second steps and let rendering happen whenever it can. The scaffold's pfLoop already does this; we just feed it an update(dt) and a render().

🧰 The engine, loaded on every page in this track

  • TileMap + drawTileMap — the level grid and its renderer (solid blocks; later, one-way platforms and slopes).
  • AABB + moveAndCollide — the player's box and the routine that moves it through the grid and stops it at solids. This is the most important function in the track.
  • pfInstallKeys + pfLoop — held-key input (with click-to-focus) and the fixed-timestep loop.

🌍 Gravity & the game loop

Every step, gravity adds a bit to the box's downward velocity, and that velocity moves the box. Two lines. Watch the box fall, speed up, and land — the fading trail shows it accelerating (the gaps grow each step). Drag the gravity slider; toggle the trail.

📝 Click to see: the entire loop is two lines of physics 
const GRAVITY = 1500;    // px/s²  — a downward acceleration
const TERMINAL = 900;    // px/s   — fastest you may fall
let vy = 0;              // px/s   — current vertical velocity

function update(dt) {            // dt is a FIXED 1/60 s, every step
  vy = Math.min(vy + GRAVITY * dt, TERMINAL);   // 1) accelerate
  moveAndCollide(box, 0, vy * dt, box.map);     // 2) move by v·dt, stop at solids
}

// pfLoop calls update() in fixed steps and render() once per frame:
pfLoop(update, render).start();

Why clamp to a terminal velocity? It keeps falls feeling controlled — but it also keeps each step's movement smaller than a tile, which is exactly what moveAndCollide needs to never tunnel through thin floors. (High-speed tunneling is a whole topic in the Simulations tier.)

🧊 Per-axis collision — the primitive

Gravity is off here so collision is the only thing happening. Fly the box around with arrows / WASD; it stops flush against any solid tile, and you can slide along a wall. The HUD shows which sides are touching — the {left, right, up, down} flags moveAndCollide returns.

click, then move…

Why one axis at a time?

moveAndCollide moves the box horizontally and resolves, then vertically and resolves — never both at once. That separation is the whole trick: "walk into a wall" and "land on a floor" stay independent, so a box sliding across a tiled floor doesn't snag on the seams between tiles.

📝 Click to see: the shape of the per-axis resolver 
// (this is engine/physics.js — you just CALL it)
function moveAndCollide(box, vx, vy, map) {
  const hit = { left:false, right:false, up:false, down:false };

  box.x += vx;                       // 1) move on X...
  if (vx > 0 && solidAhead(box.right, map)) { box.x = snapLeftOfWall();  hit.right = true; }
  if (vx < 0 && solidAhead(box.left,  map)) { box.x = snapRightOfWall(); hit.left  = true; }

  box.y += vy;                       // 2) ...THEN move on Y (using the fixed x)
  if (vy > 0 && solidBelow(box.bottom, map)) { box.y = snapOntoFloor();  hit.down = true; }
  if (vy < 0 && solidAbove(box.top,    map)) { box.y = snapBelowRoof();  hit.up   = true; }

  return hit;                        // who did we touch this step?
}
interface Hit { left: boolean; right: boolean; up: boolean; down: boolean; }

function moveAndCollide(box: AABB, vx: number, vy: number, map: TileMap): Hit {

  box.x += vx;                       // 1) move on X...

  if (vx < 0 && solidAhead(box.left,  map)) { box.x = snapRightOfWall(); hit.left  = true; }

  if (vy < 0 && solidAbove(box.top,    map)) { box.y = snapBelowRoof();  hit.up   = true; }

  return hit;
}

⬆️ Grounded & the jump

A jump is just a sudden upward velocity — but only allowed when you're standing on something. "Standing on something" is exactly hit.down from the last move, which we remember as onGround. Walk with ← →, jump with ↑ or Space. There's no double-jump — that's a deliberate choice we revisit in the next tier.

let vx = 0, vy = 0, onGround = false;

function update(dt) {
  vx = input.axisX() * MOVE_SPEED;                 // -1 / 0 / +1  →  left/right

  if (input.pressed('jump') && onGround) vy = -JUMP;   // launch — only if grounded

  vy = Math.min(vy + GRAVITY * dt, TERMINAL);      // gravity
  const hit = moveAndCollide(box, vx * dt, vy * dt, map);

  if (hit.down) vy = 0;                            // landed
  if (hit.up && vy < 0) vy = 0;                    // bonked head
  onGround = hit.down;                             // remember for next step's jump
  input.endFrame();                               // consume this step's key edges
}

Jump height is physics, not magic: a launch speed v against gravity g reaches v² / (2g) pixels high. The HUD shows it live — nudge the sliders and watch the number (and the box) agree.

🏃 Running — acceleration, max speed & friction

The same ← → input drives two boxes. The top one snaps instantly to full speed and back to zero — it feels robotic. The bottom one accelerates toward max speed and decelerates via friction when you let go — it feels alive. Same controls, completely different feel. Tune the three knobs and find a feel you like.

const ax = input.axisX();              // -1 / 0 / +1
if (ax !== 0) {
  vx += ax * ACCEL * dt;               // push toward the held direction

🎮 Mini-Project: "First Steps"

Everything at once: run (accel + friction), jump (grounded), gravity, and per-axis tilemap collision, on a hand-built level. Climb the platforms and reach the gold flag. Miss a jump into the pit and you respawn — a free reminder that off the edge of the tilemap is just empty air, so you fall off the map. ← → run, ↑ / Space jump.

function update(dt) {
  // 1. horizontal: accelerate while held, friction when released
  const ax = input.axisX();
  if (ax !== 0) vx += ax * ACCEL * dt;
  else { const d = FRICTION * dt; vx = Math.abs(vx) <= d ? 0 : vx - Math.sign(vx) * d; }
  vx = clamp(vx, -MAX_SPEED, MAX_SPEED);

  // 2. jump only when grounded
  if (input.pressed('jump') && onGround) vy = -JUMP;

  // 3. gravity, then move & resolve against the tilemap
  vy = Math.min(vy + GRAVITY * dt, TERMINAL);
  const hit = moveAndCollide(box, vx * dt, vy * dt, map);
  if (hit.down) vy = 0;
  if (hit.up && vy < 0) vy = 0;
  if (hit.left || hit.right) vx = 0;
  onGround = hit.down;

  // 4. lose (fell off) / win (touched the goal)
  if (box.top > WORLD_H + 60) respawn();
  if (box.intersects(goal)) won = true;
  input.endFrame();
}

✅ Recap — what you built

  • The fixed-timestep loop — update(dt) in 1/60 s steps, render whenever.
  • Gravity & integration — accelerate velocity, move by v · dt, clamp to terminal.
  • Per-axis AABB-vs-tilemap collision — move X & resolve, then Y; the {left,right,up,down} flags.
  • Grounded + jump — onGround = hit.down; launch only when grounded.
  • Run feel — acceleration toward max speed, friction back to zero.
