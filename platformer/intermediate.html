✨ What "game feel" actually is

Play the Beginner mini-project and then play Celeste, and the difference isn't the physics equations — they're nearly identical. The difference is a stack of small forgiveness rules and curve tweaks that quietly cover for human reaction time and make the avatar feel like an extension of your hand. None of them are hard; together they're everything.

Every demo below drives the same controller with one feature toggled, so you can feel each in isolation. The capstone turns them all on at once.

💡 The golden rule of feel: the player is never wrong. If a jump looks like it should have worked, it should have worked. Coyote time, jump buffering, and corner correction all exist to honour the player's intent, not the simulation's pixel-exact truth.

🐺 Coyote time

Named after the cartoon coyote who hangs in the air before falling: for a few milliseconds after you walk off a ledge, you can still jump. Run off the platform and try to jump a hair too late — with coyote time on (the player flashes gold during the grace window), it still launches. ← → run, ↑/Space jump.

// each step: refill the window while grounded, drain it in the air
if (onGround) coyoteLeft = COYOTE_TIME;          // e.g. 0.09 s
else          coyoteLeft = Math.max(0, coyoteLeft - dt);

// you may jump if grounded OR still inside the grace window
const canJump = onGround || coyoteLeft > 0;
if (jumpPressed && canJump) { vy = -JUMP; coyoteLeft = 0; }
if (onGround) coyoteLeft = COYOTE_TIME;
else          coyoteLeft = Math.max(0, coyoteLeft - dt);

⏱️ Jump buffering

The mirror image of coyote time. If you press jump a few frames before you land, the press is remembered and fires the instant you touch down — so an eager early press is never swallowed. Jump around and mash ↑/Space just before landing; with buffering on, the player flashes purple while a press is queued.

window 120 ms
click, then jump & mash before landing…
📝 Click to see: jump buffering is the same timer, the other way round 
// start the window when jump is pressed; drain it otherwise
if (jumpPressed) bufferLeft = BUFFER_TIME;       // e.g. 0.11 s
else             bufferLeft = Math.max(0, bufferLeft - dt);

// fire as soon as a jump is allowed AND a press is still queued
if (bufferLeft > 0 && (onGround || coyoteLeft > 0)) {
  vy = -JUMP; bufferLeft = 0; coyoteLeft = 0;
}
if (jumpPressed) bufferLeft = BUFFER_TIME;
else             bufferLeft = Math.max(0, bufferLeft - dt);

if (bufferLeft > 0 && (onGround || coyoteLeft > 0)) {
  vy = -JUMP; bufferLeft = 0; coyoteLeft = 0;
}

📏 Variable jump height

A quick tap should be a small hop; holding the button should be a full jump. The trick: when you release the button while still rising, cut the upward velocity. The two dotted lines mark the short-hop and full-jump apex heights — tap and hold to land between them. ↑/Space to jump.

click, then tap vs hold jump…
📝 Click to see: release-to-cut 
if (jumpPressed && canJump) { vy = -JUMP; jumping = true; }

// released the button while still going up? snip the rise short.
if (jumping && !jumpHeld && vy < 0) {
  vy *= CUT_FACTOR;        // e.g. 0.45 — a tap reaches ~CUT² of full height
  jumping = false;
}
if (vy >= 0) jumping = false;   // past the apex; nothing left to cut
if (jumpPressed && canJump) { vy = -JUMP; jumping = true; }

if (jumping && !jumpHeld && vy < 0) {
  vy *= CUT_FACTOR;
  jumping = false;
}
if (vy >= 0) jumping = false;

🪶 Apex hangtime, asymmetric gravity & fast-fall

Real jumps feel floaty at the top and snappy on the way down. Three tweaks get you there: make fall gravity stronger than rise gravity, reduce gravity near the apex so the peak lingers (the player flashes gold during the hang), and let the player hold ↓ to fast-fall. Jump around and watch the arc trail; tune the three gravities.

rise g 1400 fall g 2200 apex band 130
click, then jump…
📝 Click to see: choosing gravity per phase of the jump 
// pick gravity based on what the jump is doing right now
let g = (vy < 0) ? GRAV_RISE : GRAV_FALL;            // snappier descent

if (Math.abs(vy) < APEX_BAND) g *= APEX_MULT;        // float at the top (~0.55)
if (vy > 0 && downHeld)        g *= FAST_FALL_MULT;    // hold ↓ to drop fast

vy = Math.min(vy + g * dt, TERMINAL);
let g: number = (vy < 0) ? GRAV_RISE : GRAV_FALL;

if (Math.abs(vy) < APEX_BAND) g *= APEX_MULT;
if (vy > 0 && downHeld)        g *= FAST_FALL_MULT;

vy = Math.min(vy + g * dt, TERMINAL);

📐 Corner correction

You jump up through a gap, clip the corner of a ledge by two pixels, and bonk straight back down — infuriating, and never what you meant. Corner correction checks: if a small sideways nudge would have cleared the obstruction, apply it and let the jump continue. Aim the gap roughly and jump; with correction on the player flashes green when it's saved.

click, then jump through the gap…
📝 Click to see: forgive a head-bonk that a nudge would clear 
const beforeY = box.y;
const hit = moveAndCollide(box, 0, vy * dt, map);   // vertical move

if (hit.up && vy < 0) {                  // bonked the ceiling while rising
  let saved = false;
  for (let mag = 1; mag <= CORNER && !saved; mag++) {
    for (const dir of [facing, -facing]) {           // try a small sideways shift
      const t = new AABB(box.x + dir*mag, beforeY, box.w, box.h);
      if (!moveAndCollide(t, 0, vy*dt, map).up) {     // ...does it clear now?
        box.x = t.x; box.y = t.y; saved = true; break;
      }
    }
  }
  if (!saved) vy = 0;                    // a real ceiling — stop the rise
}
const beforeY = box.y;
const hit = moveAndCollide(box, 0, vy * dt, map);

if (hit.up && vy < 0) {
  let saved = false;
  for (let mag = 1; mag <= CORNER && !saved; mag++) {
    for (const dir of [facing, -facing]) {
      const t = new AABB(box.x + dir*mag, beforeY, box.w, box.h);
      if (!moveAndCollide(t, 0, vy*dt, map).up) {
        box.x = t.x; box.y = t.y; saved = true; break;
      }
    }
  }
  if (!saved) vy = 0;
}

🔁 The player state machine

Animation, sound, and effects all key off what the player is doing, so it's worth naming those states explicitly. A platformer character is a tiny finite state machine driven straight from physics — no hand-written transitions, just a read of velocity and grounded:

IDLEgrounded, ~still
RUNgrounded, moving
JUMPairborne, vy < 0
FALLairborne, vy ≥ 0
LANDjust touched down

In the Feel Lab below, the live state is printed over the player, and each state drives the squash-and-stretch (stretch on takeoff, squash on landing) — the cheapest, highest-impact piece of juice there is.

📝 Click to see: the whole FSM is five lines 
const justLanded = onGround && !wasGround;
if (justLanded) landTimer = 0.12;       // a brief landing pose

if (landTimer > 0)       { landTimer -= dt; state = 'land'; }
else if (!onGround)      state = vy < 0 ? 'jump' : 'fall';
else                     state = Math.abs(vx) > 12 ? 'run' : 'idle';
type State = 'idle' | 'run' | 'jump' | 'fall' | 'land';
